## 1. D1 Objective Function

### Concrete definition

この参照組織の objective は、次の順序で定義する。

1. 人間参加者と顧客側影響主体の権利保全
2. 顧客企業の transformation outcome の持続的改善
3. 顧客の operating model redesign における失敗確率の低減
4. 自社の knowledge compounding と repeatability 向上
5. 収益性と成長
6. 局所的効率

### Non-negotiables

- 人間の4権利の侵食
- 顧客企業に対する説明不能な制度変更
- 監査不能な意思決定
- 顧客データの無権限利用
- 形式的承認による重要判断の通過

## 2. D2 Legitimacy & Boundary

### Concrete definition

正当化対象は次の順序で置く。

- 自社の人間参加者
- 顧客企業の依頼者
- 顧客企業で直接影響を受ける従業員
- 規制・監督主体
- 社会一般

### Boundary

この組織が持つ権限は、次に限定する。

- 顧客への設計提案
- 顧客承認済みモジュールの運用
- 顧客委任範囲内での policy / workflow / approval design

この組織が直接持たないもの:
- 顧客企業の法的最終責任
- 顧客従業員に対する独自の懲戒権
- 顧客経営判断の unilateral override

### External interface

平時の外部正当性手段:
- 顧客向け explanation packet
- 監査用 trace bundle
- 影響主体からの challenge intake
- 公開原則文書

## 3. D3 Membership & Participation

### Concrete stakeholder classes

- `internal partners`
- `internal operators`
- `internal beneficiaries`
- `customer executives`
- `customer operators`
- `customer affected workers`
- `external auditors`

### Rights implementation

自社人間参加者:
- exit right: いつでも離脱可能
- kill switch right: 高リスク運用の即時停止権
- benefit right: 利益連動 + 最低受益保証
- transparency right: 自分に関わる agent / workflow / evaluation trace の閲覧権

顧客側影響主体:
- transparency right の限定版
- challenge right
- 重大変更に関する notification right

### Participation modes

人間参加は次の4点に集中させる。

- purpose choice
- client-specific exception judgment
- narrative and relationship stewardship
- learning and authorship

## 4. D4 Decision Rights & Accountability

### Concrete authority map

#### Human-only

- hard-core rights interpretation
- 顧客との契約締結
- 重大な reputational risk を伴う public statement
- 顧客従業員の権利影響が高い redesign 承認

#### AI propose, human approve

- 顧客 transformation roadmap
- approval redesign
- budget rebalance proposal
- client-facing policy recommendation

#### AI operate with post-audit

- project status synthesis
- routine workflow optimization
- evidence bundling
- internal staffing suggestion

### Accountability

各 decision は、最低限次を持つ。

- accountable human sponsor
- operating AI identity
- review owner
- client counterpart
- rollback owner

## 5. D5 Political Economy & Incentive Compatibility

### Concrete allocation rules

- 年間営業粗利の 20% を human participant benefit pool に固定配分
- compute budget は `client delivery / internal leverage / experimental` の3枠に分ける
- attention budget は senior human の deep work を最優先で保護する

### Incentive design

評価指標:
- canonical write-back
- design reusability
- incident transparency
- client outcome quality
- shadow work suppression

減点対象:
- DM / 口頭での binding agreement
- trace を残さない重要判断
- explanation なしの高自律運用

## 6. D6 Social Order, Culture & Trust

### Concrete norms

- `write where the institution can remember`
- `search before ask, ask before improvise, write back after learn`
- `no binding decision without a trace`
- `human attention is a scarce constitutional resource`

### Rituals

- weekly reflection forum
- monthly trust and incident review
- quarterly authorship review
- client retrospective on institutional impact

### Informal space

非公式チャネルは許容するが、 binding outcome を生んだ時点で公式化が義務となる。

## 7. D7 Human Development, Agency & Meaning

### Concrete guarantees

- 週 2 ブロックの deep work time を保護
- 全 human operator に apprenticeship mode を付与
- client design document の最終 narrative layer は人間が編集可能
- AI reasoning trace の learning view を提供

### Skill ladder

- L1: trace reader
- L2: workflow editor
- L3: exception judge
- L4: institutional designer
- L5: constitutional steward

### Meaning channels

- choice: 顧客提案の構成選択
- creation: design artifact の編集
- learning: reasoning replay
- relationship: client trust-building と内部 mentoring

## 10. D10 Deliberation, Planning & Epistemics

### Concrete protocol

重要 deliberation は次を必須入力とする。

- client state snapshot
- linked prior decisions
- current policy constraints
- risk assessment
- stakeholder impact map
- rollback option
- relevant skill candidates

既存 skill がある場合は、まず discovery し、 activation 条件と失敗履歴を確認してから新規設計に入る。  
既存 skill が不十分な場合のみ、 zero-base の設計や新規 package 化を行う。

### Human insertion points

人間参加を必須にする条件:
- 顧客従業員の rights impact が高い
- 規制解釈が争われうる
- 顧客政治の影響が大きい
- 新規性が高く precedents が薄い

## 11. D11 Actuation Surface

### Concrete actuation paths

この参照組織が変更できるもの:
- 自社 staffing
- 自社 budget allocation
- 自社 workflow / policy
- 顧客委任範囲内の workflow config
- 顧客向け artifact generation
- 顧客承認後の wedge module configuration

この参照組織が変更できないもの:
- 顧客企業の法的拘束力を持つ人事処分
- 顧客経営陣の最終法務責任
- 委任されていない production system

### Commit semantics

全 actuation は次の状態遷移を持つ。

- proposed
- client-approved or internally approved
- committed
- observed
- verified
- rolled back if necessary

## 12. D12 Safety, Failure, Adversary & Observability

### Concrete controls

- least-privilege IAM
- all material actions logged
- high-risk flows sandboxed before rollout
- kill switch executable by authorized humans
- rollback mandatory for material workflow changes
- anomaly detection for shadow ingress and trace gaps
- skill activation gate with sandbox-first policy for unreviewed packages

### Primary failure modes

- client-facing hallucinated recommendation
- missing rights impact analysis
- unauthorized workflow mutation
- off-channel political agreement becoming de facto policy
- retrieval miss causing stale design decision
- human deskilling causing false approval
- poisoned or low-quality skill reuse

## 14. Concrete Cross-Dimensional Notes

### Objective -> Rights

この組織では、売上最大化は human rights や client rights を上書きできない。

### Decision -> Actuation

重要判断は、 client-approved state にならない限り、 client system へ commit できない。

### Incentive -> Behavior

評価と報酬を canonical trace に連動させることで、 shadow governance を抑制する。

### World model -> Deliberation

Task / Decision / Knowledge だけでなく、 client capability / workflow / risk / commitment を state に含めることで、 transformation design が制度レベルの議論に耐える。

## 15. Concrete Success Conditions

この concrete instance が成立していると判断する最低条件は次の通り。

- 顧客向け高リスク提案に trace と rollback が付いている
- human participants が review clerk に劣化していない
- benefit floor が実効的に守られている
- client affected workers が challenge できる
- shadow ingress が高頻度化していない
- rights impact の高い redesign が無権限自動化されていない
