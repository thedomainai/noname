# Go-to-Market Strategy
## Governed Workflow 事業戦略

## 位置づけ

本ドキュメントは、`corporate-os` の設計資産を、実際に売れる事業としてどう立ち上げるかを定義する戦略文書である。

正本ではない。  
`00-core-ai-native-institution-os-complete-spec.md` が制度設計の正本であり、本書はその事業化方針を定義する。

特に以下を決める。

- 何を売るか
- 誰に売るか
- 何を捨てるか
- どの順番で拡張するか
- `corporate-os` をどこまで表に出すか

## 1. エグゼクティブサマリー

この事業の正しい戦略は、`corporate-os` をそのまま「思想」や「汎用プラットフォーム」として売ることではない。  
また、単なる prompt 集や playbook 集としての「あんちょこ事業」に留まることでもない。

採用すべき戦略は次である。

> **外向きには、特定業務に対する即効性の高い AI 実装商品を売る。**  
> **内向きには、その導入運用から得られる判断・権限・状態・監査の構造を `corporate-os` に蓄積し、Control Plane として抽象化する。**

一言で言えば、

- 入口商品は `governed workflow`
- 提供価値は `成果 + 統治`
- 内部資産は `corporate-os`
- 将来の本丸は `workflow control plane`

である。

この順序を取る理由は3つある。

1. 顧客は抽象理論ではなく、特定業務の改善に予算を払う
2. 汎用 agent platform や semantic layer は大手基盤プレーヤーに飲み込まれやすい
3. しかし重要業務における判断責任、承認、追跡、例外処理は企業固有性が高く、防御可能な価値になる

したがって、事業の基本方針は `platform-first` ではなく `wedge-first` である。  
ただし、単なる受託自動化に堕ちないために、各案件から共通化できる構造を `corporate-os` に回収し続ける。

## 2. 問い

この事業で解くべき問いは次の5つである。

1. 何を市場に売るのか
2. どの顧客セグメントから始めるのか
3. どのレイヤーに集中し、どこで戦わないのか
4. どうすれば短期売上と長期 moat を両立できるのか
5. 3年以内に、受託から製品へどう移行するのか

## 3. 市場認識

### 3.1 市場の実態

2026年時点の企業AI市場は、次の状態にある。

- 生成AIの関心は普及した
- PoC は大量に存在する
- 本番運用に進む案件は増えている
- しかし、全社的に安定稼働する運用モデルは未成熟である
- 失敗の主因はモデル性能不足より、権限、責任、例外処理、監査、業務組み込みの不備である

つまり、企業が本当に不足しているのは「もっと賢いモデル」ではなく、**AIを重要業務に載せるための運営構造**である。

### 3.2 顧客の痛み

企業側の主要な痛みは、次の4つに集約される。

- `どの業務をAIに任せてよいかわからない`
- `任せた後に、誰が責任を持つのかわからない`
- `例外時の停止・承認・巻き戻しが設計されていない`
- `成果は欲しいが、全社プラットフォーム導入は重すぎる`

この痛みは、汎用チャット導入では解決しない。  
必要なのは、特定の高価値業務において AI を実運用できるようにする `governed workflow` である。

### 3.3 競争構造

市場には大きく3種類のプレーヤーがいる。

#### A. Foundation / runtime プレーヤー

- OpenAI
- Anthropic
- Google
- Microsoft
- Snowflake
- Databricks

この群は、モデル、agent runtime、semantic layer、copilot front door を強く持つ。

彼らに対して不利な領域:

- 企業固有の権限設計
- 承認責任の埋め込み
- 部門横断の業務例外処理
- 日本企業特有の文書、稟議、監査、暗黙運用

#### B. SI / DX / コンサル群

- 大手SI
- AI導入支援会社
- 業務コンサル

この群は、導入と顧客接点は強いが、共通プロダクト資産を持ちにくい。  
案件ごとにカスタム化しやすく、知見が再利用資産になりにくい。

#### C. Prompt / playbook / テンプレ群

- 業務テンプレ
- prompt 集
- AI 活用研修

この群は導入障壁が低い反面、模倣が容易で、継続課金の根拠が弱い。

### 3.4 勝てる場所

したがって勝てる場所は、次の交点にある。

- 汎用基盤の下ではなく上
- 単なるテンプレ販売ではなく実運用の中
- SI の個別案件ではなく再利用可能な統治構造の上

事業上の狙いは、`workflow execution` と `control plane` の交点である。

## 4. 戦略的選択

### 4.1 採らない戦略

まず明確に、採らない戦略を定義する。

#### 採らない戦略1: platform-first

最初から `corporate-os` を汎用プラットフォームとして売る。

捨てる理由:

- 顧客にとって抽象度が高すぎる
- 導入単位が大きく、営業サイクルが長くなる
- 大手基盤ベンダーと比較されやすい
- 成果が出る前に組織設計コストが先に立つ

#### 採らない戦略2: cheat-sheet-only

playbook や prompt の具体解だけを量産する。

捨てる理由:

- 模倣容易性が高い
- 差別化が継続しにくい
- 成果は出ても、構造資産が蓄積しない
- 価格競争と人的稼働依存に陥りやすい

### 4.2 採る戦略

採るべき戦略は次である。

> **Wedge-first, governed workflow, hidden OS**

意味は次の通りである。

- `Wedge-first`: まずは1業務に絞る
- `Governed workflow`: AI 実行そのものではなく、実行を支える承認、責任、追跡、例外処理まで売る
- `Hidden OS`: `corporate-os` は内部資産として蓄積し、外部には段階的にしか見せない

## 5. 提供価値の定義

### 5.1 顧客に売るもの

顧客に売るのは、`AI導入` ではない。  
売るのは、**重要業務を AI で安全に回す能力**である。

初期商品の中核は次の3つで構成する。

1. `Decision Ledger`
2. `Autonomy Mode`
3. `Verified Playbooks`

### 5.2 商品要素

#### Decision Ledger

すべての重要判断について、最低限次を記録する。

- 何を判断したか
- 何を根拠にしたか
- 誰が責任者か
- 誰が影響主体か
- どの承認経路を通したか
- どこまで巻き戻せるか

#### Autonomy Mode

業務タスクごとに、AIの自律度を段階化する。

例:

- `A0`: AI は検索・整理のみ
- `A1`: AI は提案するが、人間が承認
- `A2`: AI は定型処理を実行し、事後監査
- `A3+`: 条件付きで高自律運用

この機構により、顧客は「全部任せるか、全部止めるか」の二択から脱出できる。

#### Verified Playbooks

単なる prompt 集ではなく、次を含む業務実装パッケージ。

- 良い入力例
- 良い判断例
- 失敗例
- 例外条件
- 承認条件
- 書き戻し先
- 監査方法

これにより、現場は抽象理論を学ばずとも成果を出せる。

### 5.3 顧客にとっての便益

この商品の便益は、次の4つで説明する。

- `導入しやすい`: 全社導入ではなく1業務から始められる
- `成果が見えやすい`: 時間短縮、品質改善、例外率低下を計測できる
- `事故りにくい`: 承認、trace、rollback が埋め込まれている
- `拡張しやすい`: 1業務で作った構造を次業務へ転用できる

## 6. 顧客戦略

### 6.1 Ideal Customer Profile

初期顧客は次である。

- 日本の大企業または大企業子会社
- 従業員規模 1,000-10,000 人程度
- 複数部門にまたがる承認業務を持つ
- AI推進の予算はあるが、全社標準はまだ未確立
- 現場任せのPoC乱立に課題を感じている

### 6.2 購買者

主購買者は次のいずれかである。

- COO 直下の transformation office
- CIO / CDO / CTO
- 経営企画部
- 業務改革部門

現場部門長はスポンサーになりうるが、初期の主購買者としては弱い。  
理由は、この商品が業務改善と統治設計の両方を含むためである。

### 6.3 初期ユースケース

最初の wedge として選ぶべき業務は、次の条件を満たすものに限定する。

- 高頻度である
- 判断や承認が絡む
- 例外処理が多い
- ログと成果を計測しやすい
- 既存フローが重い

候補:

- 稟議・例外承認
- 変更管理
- 提案レビュー / 設計レビュー
- 顧客対応の高リスクトリアージ
- 内部監査用の evidence bundling

最初の1業務は、`ナレッジ検索` のような汎用横断テーマではなく、**判断責任が明確な業務**を選ぶべきである。

## 7. GTM モデル

### 7.1 営業の基本設計

営業モデルは次の3段階にする。

1. `診断`
2. `wedge 導入`
3. `年間運用 + 拡張`

### 7.2 商品パッケージ

#### Phase 1: Workflow Readiness 診断

期間:

- 4週間

内容:

- 対象業務の可視化
- AI適用余地の判定
- 判断責任と承認構造の棚卸し
- shadow workflow の抽出
- 初期 wedge 設計

価格感:

- 300万-500万円

#### Phase 2: Wedge 導入

期間:

- 8-12週間

内容:

- Decision Ledger 実装
- Autonomy Mode 設計
- Verified Playbooks 作成
- KPI 設計
- 現場導入

価格感:

- 1,000万-2,000万円

#### Phase 3: Managed Governance

期間:

- 年間契約

内容:

- 月次レビュー
- autonomy threshold の調整
- 例外分析
- trace / incident 監査
- 次 workflow への展開支援

価格感:

- 年額 1,200万-3,600万円

### 7.3 売り方

言い方を間違えると失注する。  
初期の販売メッセージは、次に限定する。

- `AIを安全に現場導入する`
- `重要業務に AI を載せられるようにする`
- `任せてよい範囲を可視化する`
- `承認と責任の設計まで含めて導入する`

避けるべき表現:

- `管理職を置き換える`
- `会社のOSを入れ替える`
- `汎用AIプラットフォーム`
- `次世代semantic layer`

## 8. プロダクト戦略

### 8.1 プロダクトの核

初期プロダクトで絶対に持つべき核は次の通りである。

- decision object
- authority map
- autonomy rules
- provenance log
- rollback path
- exception queue
- policy surface

### 8.2 プロダクト化の順序

#### ステップ1: サービス主導

初期はサービス色が強くてよい。  
ただし、毎回ゼロから作ってはならない。

共通化対象:

- 業務オブジェクトの型
- 承認パターン
- autonomy 判定ロジック
- trace schema
- audit output

#### ステップ2: 半製品化

3社程度の導入で共通構造が見えたら、次を製品化する。

- policy engine
- decision ledger UI
- workflow state model
- exception operations console

#### ステップ3: Workflow Control Plane 化

複数 workflow にまたがる共通制御面として、次を提供する。

- workflow ごとの autonomy 管理
- cross-workflow policy 管理
- organizational traceability
- audit bundle export

### 8.3 `corporate-os` の扱い

`corporate-os` は当面、顧客にフルパッケージで見せるものではない。  
役割は次の3つである。

- 案件から学んだ構造を回収する内部憲法
- プロダクト要件を外さないための完全性地図
- 将来、カテゴリ物語を立ち上げるための知的資産

つまり、最初の3年間は `visible product < hidden system` の構図を保つ。

## 9. 経済性

### 9.1 単価構造

この事業は、初期はソフトウェア単体課金より、`設計 + 導入 + 運用` の束で売るべきである。

理由:

- 顧客は製品カテゴリにまだ予算を置きにくい
- 実装責任を伴う方が発注しやすい
- 初期はプロダクトより deployment quality の差が成果を左右する

### 9.2 利益構造

理想構造は次である。

- Year 1: サービス粗利で顧客獲得
- Year 2: 運用とライセンスで粗利改善
- Year 3: workflow 拡張で NRR を上げる

このため、LTV の源泉は新規案件数より `2 workflow 目、3 workflow 目への展開` に置くべきである。

### 9.3 避けるべき収益構造

- prompt 作成工数の時間売り
- 個別業務ごとの完全受託
- 安価な研修売り切り
- 汎用チャット導入支援のみ

これらは売上にはなるが、Control Plane への資産化が弱い。

## 10. 3年ロードマップ

### 10.1 0-12か月

目的:

- wedge の成立
- 3社の lighthouse 顧客獲得
- 1業務での成果再現

やること:

- 対象業務を1つに固定
- playbook と trace schema を標準化
- autonomy level 定義を固定
- decision ledger の最小プロダクトを作る

成功基準:

- 3社導入
- Time to first value 45日以内
- 主要KPI改善を顧客と合意

### 10.2 12-24か月

目的:

- 受託から半製品へ移行
- 1業務内で repeatability を証明

やること:

- 共通 UI と policy engine を整備
- 業界共通テンプレを形成
- Managed Governance の更新サイクルを定義

成功基準:

- 共通機能比率 50%以上
- 2 workflow 目展開の成功
- ライセンス売上比率の上昇

### 10.3 24-36か月

目的:

- governed workflow company から workflow control plane company へ進化

やること:

- 複数 workflow をまたぐ可視化
- cross-workflow policy 管理
- 業界別の標準 workflow パッケージ化

成功基準:

- 既存顧客で 3 workflow 以上に展開
- NRR 120%以上
- プロダクト売上がサービス売上に対して優位化

- `1顧客あたり workflow 数`

- `サービス売上に対するライセンス売上比率`

## 12. 組織戦略

初期チームは大きくしすぎてはならない。  
必要なのは、次の4機能である。

- `workflow strategist`
- `institutional designer`
- `AI implementation lead`
- `customer success / governance ops`

重要なのは、単なる ML チームでも、単なる業務コンサルチームでも不足することだ。  
この事業は、業務設計、制度設計、実装、運用の4つを横断できる少人数精鋭で始める必要がある。

## 13. 主要リスク

### リスク1: 受託化

顧客ごとの要求に引っ張られ、構造資産が積み上がらない。

対策:

- 対象業務を絞る
- 毎案件で共通化レビューを実施する
- 再利用されない要件は原則断る

### リスク2: 抽象化先行

`corporate-os` の思想整理ばかり進み、顧客成果が遅れる。

対策:

- まず workflow 単位で売る
- 仕様ではなく導入KPIで進捗を測る

### リスク3: 安売りテンプレ事業化

playbook が単なるテンプレ集に見え、価格が維持できない。

対策:

- 必ず governance 機能と束で売る
- audit と autonomy 設計を外さない

### リスク4: 大手との正面衝突

agent platform や semantic layer のような一般カテゴリに出ていく。

対策:

- 自社カテゴリを `governed workflow` に固定する
- front door 競争には乗らない

## 14. 最終結論

この事業の本質は、`corporate-os` を売ることではない。  
また、`あんちょこ` を売ることでもない。

正確には、

> **顧客には、成果の出る governed workflow を売り、**  
> **自社の中では、その導入運用から corporate-os を育てる。**

この二層構造こそが、短期売上と長期競争優位を両立する唯一の戦略である。

したがって、経営判断としては次を採用する。

- 表の事業カテゴリは `governed workflow`
- 初期商品は `Decision Ledger + Autonomy Mode + Verified Playbooks`
- 初期顧客は日本の大企業の変革責任者
- 最初の wedge は承認責任が明確な1業務
- `corporate-os` は内部憲法として蓄積し、段階的に Control Plane へ抽象化する

この戦略のキーワードは次の一文に尽きる。

> **入口は具体、提供価値は統治、蓄積資産は OS。**
