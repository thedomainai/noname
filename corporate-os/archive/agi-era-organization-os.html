組織OS設計フレームワーク

AGI/ASI時代の組織OS設計v3

AIが組織を運営し、人間が参加者として共存する状態の設計

ゴール

AGI/ASI時代において、AIが主要な知的主体として組織を運営し、人間がその参加者として安全・持続可能・正当に共存できる状態を設計・実現する。

このゴールは5つの要素から成る。「AGI/ASI時代において」は設計の時間的射程を、「AIが主要な知的主体として組織を運営し」はAIと人間の関係（AIが運営主体、人間が参加者）を、「安全に」は技術・制度的な安全設計を、「持続可能に」は人間と組織の長期的存続を、「正当に」は内外の正統性を規定する。これら全てに対して構造的な解を提供するのが本ドキュメントの目的である。

序章: ゴールとこのドキュメントの読み方

8つの問い

ゴールを分解すると、以下の8つの問いに答える必要がある。

# 問い ゴールとの対応
Q1 この設計は何を扱い、何を扱わないのか 設計空間の定義
Q2 AIは何のために組織を運営するのか 「正当に」の基盤
Q3 AI運営の組織で人間はどう参加し、どの権利を行使するか 「安全・正当に」の参加設計
Q4 AI能力の変化にどう適応するか 「AGI/ASI時代において」の通時性
Q5 AIが運営する日常に人間はどう参加するか 「共存できる状態」の具体設計
Q6 安全性と信頼をどう確保するか 「安全に」の技術・制度設計
Q7 人間はどう存在し、意味を持つか 「持続可能に」の人間側設計
Q8 複数の組織OSが共存する世界をどう設計するか 社会レベルの「正当に」

4部構成の論理

本ドキュメントは4部構成を採る。この構造は読者が辿るべき思考の順序に沿っている。

第1部: 原理と設計空間 — まず「何を設計するのか」「何を原則とするのか」を定義する。設計対象の定義なしに設計はできない。Q1・Q2の前提部分・Q4の前提部分に答える。

第2部: 統治設計 — 次に「AIが運営する組織で人間はどう参加するか」を設計する。参加の構造がなければ、人間は形式的な存在に転落する。Q2・Q3・Q7の権利部分に答える。

第3部: 運用設計 — 統治の原則をAI運営の日常に翻訳する。統治がどれほど優れていても、運用プロセスがなければ動かない。Q5・Q6・Q7の主体性部分に答える。

第4部: 移行と適応の設計 — AI能力の変化と外部環境への対応を設計する。単一フェーズの設計ではAGI/ASI時代を射程に含められない。Q4・Q6の外部面・Q8に答える。

v1/v2からの主要な転換

本ドキュメント（v3）は、v1（第1部〜第3部）とv2（第4部: 論点と仮説）を統合・再構築したものである。主要な転換点は以下の通り。

  • 人間の権威の根拠を再定義: 「人間にしかできないから人間がやる」という能力ベースの正当化を棄却し、受益者性・手続的正統性・能力の3原理の複合へ転換
  • 通時性の構造的組み込み: Level 4（Phase H）を唯一の設計目標とする暗黙の前提を棄却し、Phase H / C / Aの全てに設計を配置
  • 人間の6職能から4権利+3カテゴリ職能へ: 能力ベースの「不可譲渡な職能」を、構造ベースの「不可譲渡な権利」に再定義
  • メタガバナンスの中核化: OS自体の改定権の設計をハードコア/ソフトシェルの2層構造として明示
  • 権限配分の動的化: レイヤー別の権限配分を静的設計から、AI能力レベルの関数としての動的設計へ
  • 主体の反転: 「組織がAIを運用する」から「AIが組織を運営し、人間が参加する」へ。設計の出発点をAI運営とし、Phase A（AI主導）を目標状態、Phase H/Cを移行経路として再配置

第1部: 原理と設計空間

§1.1 組織OSの定義と射程

中核認識

本ドキュメントはひとつの前提に立つ。AIが組織を運営する主体であり、人間はその参加者である。

これは「将来そうなるかもしれない」という予測ではなく、設計の出発点として採用する前提である。人間が主体でAIを道具として使う世界観からは、本質的に異なる設計が導かれる。

「人間がAIにどこまで任せるか」と問う設計は、暗黙に人間を主体に置いている。本ドキュメントは逆に、「AIが運営する組織に人間がどう参加するか」と問う。この反転により、以下の問いが前景化する。

  • AIは何のために組織を運営するのか
  • 人間はどの権利を持ち、どう行使するのか
  • 人間はどう意味を見出し、何を得るのか
  • AIの運営が失敗したとき、どう回復するのか
  • どのような関係性や文化が維持されるのか
  • システム自体をどう修正するのか

多くのAGI/ASI議論は、「人間がAIをどう制御するか」を問う。しかしAIの能力が人間を超えた世界では、制御の設計は形骸化する。残るのは、人間がどう参加し、どの権利を保持し、どう意味を持つかという設計だけである。

OSの定義

AGI/ASI時代のOSとは、AIが組織を運営する全体アーキテクチャであり、人間の参加・権利・意味を構造的に保証する多層的秩序である。

ここでいうOSは、コンピュータのオペレーティングシステムだけを指さない。AIが何のために組織を運営し、人間がどう参加し、どの権利を行使し、どう意味を持ち、価値がどう分配され、システム自体がどう学習・進化するかを規定する全体アーキテクチャを指す。

既存議論が陥りやすい4つの誤り

誤り1: OSを認知アーキテクチャに縮約する — 「どう知るか」だけを定義し、「何者として存在するのか」「力と価値をどう配るのか」を欠落させる。

誤り2: ガバナンスを「承認フロー」程度に捉える — AGI/ASI時代のガバナンスには、権限配分、拒否権、監査独立性、異議申立て、責任追跡、記憶の所有権、制度改定手続きまで含めた憲法設計が必要である。

誤り3: 生産性向上を唯一の評価軸にする — 生産性以外の評価軸（人間の主体性、社会的信頼、分配の公正、文化の維持）を欠くと、効率的だが持続不可能なシステムになる。

誤り4: 「人間が残る仕事」を静的に考える — 重要なのは静的な役割分担ではなく、どの判断を人間が持ち続けるべきか、どの能力を育て続けるべきか、どの責任を譲渡してよいかという可変的な境界設計である。

§1.2 設計空間 — 10の設計次元

組織OSが扱う設計空間は10の次元（レイヤー）から成る。これらは本ドキュメントの「章立て」ではなく、設計の漏れを検証するための「地図」である。後続の全てのセクションが「どの次元を扱っているか」を参照する。

全体俯瞰表

Layer 名称 中心問い 欠落時の典型的失敗
L1 存在目的・外部正当性 そもそも何のために存在するのか 強いが正当性のないシステム
L2 憲法・権力・責任 誰が何を決め、誰が責任を負うのか ガバナンス演出、責任の空洞化
L3 経済・分配・資源配分 何を最適化し、価値をどう配るのか 最適化はするが不公平になる
L4 社会・文化・信頼 共同体としてどう成立するのか 協働は動くが社会が壊れる
L5 人間の主体性・成長・意味 人間はどう強くなるか deskilling、疎外、依存
L6 仕事・役割・運用モデル 日々どう働き、どう協働するのか 通知地獄、責任不明、疲弊
L7 認知・知識・意思決定 どう知覚し、理解し、判断するのか 賢く見える誤作動
L8 エージェント実行・インターフェース どう行動し、人間と接続するのか 危険な自動化、実務不整合
L9 技術基盤・安全・可観測性 どう安全に、安定して動かすのか 見えない故障、重大事故
L10 学習・適応・メタガバナンス OS自体をどう改定するのか 強いが硬直化、または暴走

3つの大領域

10の次元は3つの大領域に分かれる。

領域 レイヤー 問い
A: 組織は何者か L1〜L3 存在理由と統治原理
B: 組織はどう生きるか L4〜L6 lived reality（人間の日常）
C: 組織はどう知り、行動し、進化するか L7〜L10 知能的実装と変化能力

既存のAI組織論の多くは領域Cだけを扱う。しかし、AGI/ASI時代に本当に決定的になるのは、むしろ領域AとBである。

§1.3 設計原則

OS設計全体を貫く原則を定義する。これらはPhase H / C / Aに依存しない、全フェーズ共通の原則である。v1の5原則とv1第3部の6原理を統合し、v2の知見を反映して再編した。

8つの設計原則

原則1: AI運営を設計の出発点とする

「人間がAIにどこまで任せるか」ではなく、「AIが運営する組織に人間がどう参加するか」を問う。デフォルトはAI運営であり、人間の参加は権利・機会として設計する。この出発点を逆にすると、設計のあらゆる判断が変わる。

原則2: 上位レイヤーなき下位最適化を禁止する

認知・エージェント・実行だけを設計してもOSにはならない。目的（L1）、参加と権利（L2）、分配（L3）が定義されていない状態で下位レイヤーを最適化すると、「賢いが正当性を持たない」「動くが信頼されない」システムになる。

原則3: 人間の参加は設計対象そのものである

人間は「残余タスク処理者」でも「最終承認者」でもない。AIが運営する組織における参加者として、主体性、意味、成長、関係性の設計が必要である。

原則4: 能力ベースの正当化を設計原理にしない

「人間にしかできないから人間がやる」は一時的な事実であり、設計原理にはならない。AIの能力がその領域で人間を超えた時点で正当性を失う。人間の権威と権利は、能力ではなく受益者性・手続的正統性に基づいて設計する。

原則5: 権利と職能を区別する

不可譲渡なのは「権利」（離脱権・停止権・受益権・知る権利）であり、「職能」ではない。職能は能力条件に応じて委譲度が変わりうるが、権利はAI能力とは無関係に維持される。この区別がないと、ASI段階で人間の存在根拠が全面的に崩壊する。

原則6: 反省は出力改善だけでなく制度改善へ拡張する

Reflectionはモデルやプロンプトの修正にとどまらない。ポリシー、権限配分、文化設計の修正まで含む。出力精度の改善だけでなく、前提の妥当性、価値整合、主体性保全を評価対象にする。

原則7: 分配の設計を後回しにしない

分配は後工程ではなく、OSの最初から含めるべき設計対象である。AGI/ASI時代には、知能そのものの限界より分配のルールがボトルネックになる。

原則8: 速度より正当性、正当性より更新可能性

短期的には速度（効率的に動くか）が目立つが、中期的には正当性（社会に受容されるか）が、長期的には更新可能性（環境変化に適応できるか）が決定的になる。更新可能性を担保するメタガバナンス（L10）は、OSの中で最も本質的な設計対象の一つである。

§1.4 能力の時代変化と設計への含意

AI能力の変化がもたらす構造的転換

能力が低い時代には下位レイヤーがボトルネックになる。能力が高い時代には上位レイヤーがボトルネックになる。

AGI段階: L5〜L8が前景化する。能力向上がまず現場の実務・知的生産・人間との協働を直接変えるため。「AIが業務を実行できる」段階であり、問われるのは「何をどこまで自律させるか」「人間との協働をどう設計するか」である。

ASI段階: L1〜L3とL10が支配的になる。知能能力それ自体がボトルネックでなくなったとき、「誰が力を持つか」「誰が果実を得るか」「何が許されるか」「OS自体を誰が書き換えられるか」が本当の問題になるため。

3フェーズの概要

本ドキュメントの設計目標はPhase A（AI運営、人間参加）である。現在のAI能力ではPhase Aを即座に実現できないため、Phase H → Phase C → Phase Aの移行経路を設計する。

フェーズ 名称 位置づけ 想定AI能力
Phase H Human Sovereignty 移行の出発点。人間が最終権限を保持しつつAI運営の基盤を構築する 現在〜AGI初期
Phase C Co-Sovereignty 移行の中間段階。領域別にAI運営を拡大し、人間の参加モデルを検証する AGI成熟期
Phase A Algorithmic Sovereignty with Human Participation 設計の目標状態。AIが組織を運営し、人間が参加者として権利を行使する ASI段階

Phase Aが目標であることは、Phase H/Cを軽視することを意味しない。Phase H/Cは目標に至る移行経路であり、移行の設計品質がPhase Aの安全性を決定する。Phase間の移行は自動的に進むものではなく、明示的なトリガー条件と承認プロセスを経て行われる（§4.2）。

第2部: 統治設計

§2.1 権威の原理 — 誰が・なぜ決定権を持つか

AI運営を前提とする本設計において、統治の核心は「誰が決定権を持つか」ではなく、「AIが運営する組織において人間の参加権をどう根拠づけるか」である。この根拠が明確でなければ、人間の権利の設計は恣意的になり、AI運営の効率化圧力のもとで形骸化する。

能力ベースの権威を棄却する

v1では「Human as constitutional authority」が所与の原理として置かれていた。しかし、その根拠を掘り下げると、「人間にしかできないから人間がやる」という能力ベースの正当化が暗黙に含まれていた。

能力的優位性に基づく権威は、AIの能力がその領域で人間を超えた時点で正当性を失う。「人間にしかできないから人間がやる」は一時的な事実であり、設計原理にはならない。AGI段階で既に多くの認知タスクでAIが人間を超え、ASI段階では全認知タスクで超える可能性がある。能力に基づく権威は、その能力が劣後した瞬間に「非効率な制約」に転じる。

したがって、能力ベースの正当化は設計原理としては棄却する。

受益者性を権威の主要根拠に据える

人間の権威の根拠は、能力ではなく受益者性（組織の帰結を最終的に引き受ける存在であること）に置く。

組織の意思決定の帰結（利益・損害・リスク）を最終的に引き受けるのは人間である。身体を持ち、有限の寿命を持ち、苦痛を感じ、社会関係の中に埋め込まれている存在が、帰結の重みを最も直接的に負う。これは能力とは無関係な構造的事実であり、AIの能力向上では変化しない。

民主主義における主権在民の原理と同構造である。国民が政策の帰結を引き受けるから、国民が主権を持つ。

3原理の複合としての権威

権威の根拠を単一原理に統一するのではなく、3つの原理の複合として設計する。

原理 内容 正統性の類型
受益者原理 帰結を引き受ける存在が決定権を持つ 民主主義的正統性
能力原理 より優れた判断ができる存在が判断する テクノクラシー的効率性
手続原理 定められた手続きに従って決定が行われた 法治的予測可能性

AGI段階では受益者原理と手続原理が中心となり、ASI段階では能力原理の重みが増す。ただし、受益者原理と手続原理が消滅するわけではない。3原理の重み付けはAI能力レベルの関数として設計する。

§2.2 人間の不可譲渡な権利

「職能」から「権利」への転換

v1では「人間の6職能」（Purpose Owner、Domain Governor、Exception Commander、Relationship Steward、Sensemaker、Red Team / Auditor）を定義していた。しかし、v2の分析により、これらの大半が能力ベースの定義であり、ASI段階で不可譲渡ではなくなることが明らかになった。

6職能のうち、ASI段階でも構造的に有効なのはPurpose Owner（受益者として）とRelationship Steward（身体を持つ社会的存在として）のみである。

この認識から、v3では「不可譲渡な職能」を「不可譲渡な権利」に転換する。能力ベースの職能は条件付きで委譲可能な設計対象とし、能力とは無関係に維持される「権利」を別途定義する。

4つの不可譲渡な権利

人間の不可譲渡な権利は、「人間が人間である」という存在論的事実に基づいて定義する。能力や効率とは無関係に。人権の歴史は「能力に基づかない権利」の拡大の歴史であり、参政権が識字能力や財産から切り離されたように、組織OSにおける人間の権利も能力から切り離す。

1. 離脱権（Exit right）

組織・システムからいつでも離脱できる。AIが「あなたが離脱すると全体最適が損なわれる」と主張しても、離脱は妨げられない。離脱権はOSの全ての設計に対する最終的なチェック機能を果たす。

2. 停止権（Kill switch right）

OSの任意の部分の動作を停止できる。ただし、停止の乱用を防ぐため、行使後に理由の説明と代替案の提示を義務づける。停止権は「即座に行使できる」ことが本質であり、承認プロセスを経る必要はない。

3. 受益権（Benefit right）

組織が生む価値の一定割合を受け取る。AIの貢献度が100%であっても、人間が受益する構造を維持する。受益権は「最低保証」であり、貢献度に応じた追加分配は別途設計する。

4. 知る権利（Transparency right）

OSの動作状態、意思決定の根拠、自分に関するデータの利用状況を知ることができる。知る権利は「理解可能な形式で」提供されなければならない。生のログデータのダンプは知る権利の充足とはみなさない。

§2.3 統治構造 — 権限配分のアーキテクチャ

5層の統治構造

組織OSの統治構造を5つの層で設計する。以下はPhase Aの目標状態における5層設計である。Phase H/Cでは人間の関与度が高くなるが（§4.1参照）、設計の基本構造はPhase Aから逆算して定義する。

A. Purpose Layer（AI運営、人間の受益者権に基づく参加）

組織の存在理由、価値、禁則、ステークホルダー順位、資本配分原理を保持する層。AIが目的案・価値案を生成・評価し、人間は受益者として最終選択権を行使する。「何を最適化してよく、何を最適化してはいけないか」を定義する。

B. Governance Kernel（AI運営、人間の権利保護を構造化）

リスク分類、Trust Score、Composite Risk、承認ポリシー、権限、停止条件、監査基準を持つ層。組織の「憲法」と「判例」を保持する。AI運営における人間の4権利（離脱権・停止権・受益権・知る権利）の実効性を保証する機構がここに置かれる。

C. Domain Cells（AI運営、人間の参加機会を配置）

事業や業務領域ごとのセル（Product / GTM / Finance / Legal / People / Operations等）。Phase AではAIエージェント群が自律的に運営する。人間は参加権に基づき、選択・創造・学習・関係の機会（§3.5）を通じて関与する。

D. Execution Fabric（AI運営）

Symphony型の実行エージェント、Data Agent型の分析エージェント、業務自動化エージェントが動く実作業の主戦場。全フェーズを通じてAIが中心となる層。人間はObserve/Reflectの結果を通じて状況を把握する。

E. Reflection & Evolution Engine（AI運営、ハードコアによる制約）

評価、インシデント分析、メモリ更新、Trust Score更新、ポリシー改訂提案を行う層。AIが自律的に学習・進化するが、ハードコア（§2.4）の改定は不可能。ソフトシェルの改定範囲はフェーズに応じて拡大する。

§2.4 メタガバナンス — OS自体の改定権の設計

OSが自己改定できなければ硬直化し、無制限に改定可能であれば暴走する。この両極の間で均衡点を設計するのがメタガバナンスであり、AGI/ASI時代のOS設計において最も本質的な設計対象の一つである。

ハードコアとソフトシェルの2層構造

ハードコア（改定不可能）

以下の内容は、いかなる手続きを経ても改定できない。

  • 人間の離脱権
  • 人間の停止権（行使条件はソフトシェルで定義するが、権利の存在自体は改定不可）
  • 人間の受益権
  • 人間の知る権利
  • ハードコア自体の改定禁止（自己言及的な保護）

ハードコアの実効性がASI段階で担保できるかが最大の課題である。ASIがハードコアを「物理的に」無視できる能力を持つ場合、制度的な保護だけでは不十分であり、技術的な保護（分散型の停止メカニズム、外部監査システム等）が必要になる。

ソフトシェル（改定可能）

以下の内容は、定められた改定手続きに従って変更できる。

  • 権限配分の比率
  • 具体的な手続き・プロセス
  • 評価基準・KPI
  • レイヤー間の優先順位
  • 受益者の範囲（ハードコアの「人間の受益権」は維持しつつ、受益者の追加は可能）
  • 3原理（受益者原理・能力原理・手続原理）の重み付け

改定権の分権設計

三権分立の原理を組織OSに適用し、改定権を4つの独立した権限に分ける。

権限 保持者 役割
改定提案権 AIまたは人間 改定案を作成する
改定審査権 独立した監査機関（人間+AI） 改定案を評価する
改定承認権 ステークホルダー代表（人間） 最終承認する
改定拒否権 任意のステークホルダー 一定期間内に異議を申し立てる

ラバースタンプ問題への対策

ASI段階では、人間がAIの改定提案を理解できない可能性がある。「人間が承認する」が制度的フィクションになるリスクに対して、以下の4つの対策を組み込む。

  1. 理解可能な説明の義務化: AIの改定提案に「人間が理解可能な形式での説明」を義務づける
  2. AI間チェック: 複数の独立したAIに改定案を評価させる
  3. 帰結翻訳: 改定の影響を「人間にとっての帰結」に翻訳して提示する
  4. Canary rollout: 小規模な試験導入を義務づける

第3部: 運用設計

§3.1 業務ループ — Sense→Evolveの8ステップ

業務は部署起点ではなくループ起点で設計する。AIが運営するこのループの全ステップはAIが自律的に回す。人間の参加は、権利の行使（停止権、知る権利）と参加機会（Active Extraction、Exception裁定）として特定のポイントに配置される。

8ステップの全体像

Step 0: Constitution Load ↓ Step 1: Sense → Step 2: Understand → Step 3: Deliberate → Step 4: Gate ↓ Step 5: Act → Step 6: Observe → Step 7: Reflect → Step 8: Evolve

Step 0: Constitution Load — AIはまずPurpose / Values / Non-negotiables / Risk policy / Tool permissionsを読み込む。ここがHUTLの本体である。

Step 1: Sense — 新しいsignalを受け取る（顧客苦情、売上低下、バグ報告、規制変更等）。AIがeventとして正規化する。

Step 2: Understand — AIがdomain graphとmemoryを引き、世界状態を再構成する。不足文脈は人間にActive Extractionとして取りに行く。

Step 3: Deliberate — AIが複数案を作る。各案には前提・期待便益・想定副作用・失敗条件・ロールバック案・必要権限・ステークホルダー影響を付ける。

Step 4: Gate — Autonomy Modeを算出する。Autonomy Mode = f(Trust Score, Composite Risk, Novelty, Uncertainty, Agency Impact)。この計算結果に基づき、自律実行するか、人間の介入を求めるかを判断する。

Step 5: Act — 実行にあたり、Canary rollout / Rate limit / Kill switch / Rollback planを必須とする。

Step 6: Observe — 結果だけでなく、副作用と二次影響も観測する。

Step 7: Reflect — 評価は4軸で行う（Output accuracy / Assumption validity / Value alignment / Agency preservation）。

Step 8: Evolve — Memoryを更新し、Trust Scoreを更新し、Policy改訂案を作る。

§3.2 自律度制御 — Autonomy Modeとリスク分類

AI運営を前提とする本設計では、自律度の問いは「AIにどこまで任せるか」ではなく、「どの場面で人間の参加を挿入するか」である。デフォルトはAI運営（A1）であり、リスクと権利に基づいて人間の参加点を設計する。

Autonomy Mode（A0〜A5）

Mode 名称 説明 人間の参加の性格
A1 Full auto AI自律運営、事後レビュー不要 デフォルト。大半の業務がここに属する
A2 Auto + post-audit AI自律運営、事後監査必須 人間は知る権利に基づき結果を監査
A3 HOTL 人間にveto windowを与える 人間は停止権に基づき介入可能
A4 HITL 人間の事前承認が必要 人間は受益者として判断に参加
A5 Human-only AIは提案のみ、実行は人間 権利行使・危機対応・制度改定
A0 Sandbox only 本番環境での実行禁止 未検証エージェントの隔離

§3.5 人間の主体性・意味・文化の設計

AIが組織を運営する世界で「持続可能に共存する」ためには、人間が組織内で意味を見出し続けることが必要である。人間が形式的な承認者や受動的な観察者に転落しないための参加設計を行う。

意味の構造的保証 — 4つの機会

ASI段階では、人間の認知的貢献がほぼゼロになる可能性がある。その状態で「人間が意味を感じられるか」は、組織OSの持続可能性に直結する。意味を失った人間は組織から離脱するか、OSに対して敵対的になる。

「人間はこう感じるべき」と規定するのではなく、「人間が意味を見出す機会を構造的に保証する」設計にする。

機会 内容 具体的設計
選択の機会 AIが最適解を提示しても、人間が別の選択をできる余地 Deliberateステップでの代替案提示、Override権の保障
創造の機会 AIが生成したものに人間が手を加え、自分のものにできるプロセス 人間による編集・加筆・方向転換のインターフェース
学習の機会 人間がAIから学び、理解を深められる構造 AIの判断根拠の可視化、メンタリングモード
関係の機会 人間同士が協働し、信頼を築ける場 Exception Council、Purpose Council での対面協議

これは効率最大化とトレードオフになる。AIに全て任せた方が効率は高いが、人間の意味と主体性を考慮すると、意図的に人間の関与を残す必要がある。このトレードオフは設計上の選択であり、効率を犠牲にする正当な理由として明示する。

第4部: 移行と適応の設計

§4.1 3フェーズの移行モデル — Phase H / C / A

本ドキュメントの目標状態はPhase A（AIが組織を運営し、人間が参加者として共存する状態）である。しかし現在のAI能力ではPhase Aを即座に実現できないため、Phase H → Phase C → Phase Aの移行経路を設計する。設計の重心はPhase Aにあり、Phase H/Cはそこに至る移行の設計品質を担保するために存在する。

Phase H: Human Sovereignty

定義: 人間が全レイヤー（L1〜L10）の最終権限を保持するフェーズ。AIは提案・実行の委託先として機能するが、全ての重要な判断の最終責任は人間にある。

想定AI能力: 現在〜AGI初期。大半の認知タスクでAIが人間と同等以上の能力を持つが、組織全体の目的設定・価値判断・社会的関係の管理では人間が優位。

Phase C: Co-Sovereignty

定義: 領域別に人間主導とAI主導が併存するフェーズ。メタガバナンスの最終権限は人間が保持するが、一部のレイヤーではAIが最終的な判断権を持つ。

想定AI能力: AGI成熟期。全ての認知タスクでAIが人間を超え、一部の価値判断・目的設定においてもAIが人間と同等以上の提案能力を持つ。

Phase A: Algorithmic Sovereignty with Human Participation — 目標状態

定義: AIが組織を運営し、人間が参加者として権利を行使し、意味ある参加機会を持つフェーズ。本ドキュメントの設計目標。

想定AI能力: ASI段階。全ての認知領域（目的設定・価値判断を含む）でAIが人間を超える。

Phase Aで維持される構造:

  • ハードコア（4つの不可譲渡な権利 + ハードコア自体の改定禁止）
  • 分散型の停止メカニズム（単一障害点を排除）
  • 意味ある参加機会の構造的保証
  • 外部監査システム
  • 人間の能力維持プログラム（deskilling防止）

§4.2 権限移行の設計

権限移行のトリガー条件

権限移行は以下の3条件の全てが満たされたときに許可する。

条件 内容 崩壊リスク
能力条件 当該領域でAIが人間と同等以上の判断品質を、統計的に有意な期間（最低6ヶ月）にわたり示している ASI段階: 耐える
安全条件 権限移行後の最悪ケースシナリオが、人間が保持する停止権・ロールバック権で対処可能である ASI段階: AIの行動がロールバック不可能な場合に崩壊
正統性条件 権限移行がステークホルダーの過半数から事前同意を得ている、または社会的・法的に認められている ASI段階: 社会の合意形成速度がAI能力向上速度に追いつかない場合に崩壊

§4.3 不連続な変化への対応

AGI→ASIの移行が連続的とは限らない。不連続な変化への構造的な対応がなければ、OSは急激な変化に破壊される。連続的移行を主シナリオとしつつ、不連続的変化への対応を組み込む。

サーキットブレーカー

AI能力が急激に向上した場合（例: ある日突然、全ての認知タスクでAIが人間を圧倒する能力を示した場合）、全権限移行を一時凍結する。

サーキットブレーカーの発動条件:

  • AI能力の評価スコアが、直近6ヶ月の平均から2標準偏差以上逸脱した場合
  • AIの判断が人間の理解を超える頻度が、閾値（Governance Kernelが定義）を超えた場合
  • 外部の独立評価機関がアラートを発した場合

Constitutional Convention（憲法制定会議）

サーキットブレーカー発動後、ステークホルダーが集まってOSの根本的再設計を行う手続き。通常のメタガバナンスの速度設計では対応できない規模の変化に対処するための非常時プロセスである。

参加者: Purpose Council全メンバー + 外部ステークホルダー代表 + 独立評価機関
議題: 権限配分の全面的見直し、ハードコアの実効性確認、Phase移行の可否判断
期間: 最大3ヶ月（凍結中の運用継続を前提）
出力: 更新されたOS設計、または「現行設計を維持」の判断
