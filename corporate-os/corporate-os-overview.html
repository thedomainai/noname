中核の問い

人間より優れた知能が存在する世界において、組織はどのように運営され、駆動されるべきか。Corporate OSは、この問いに対して構造的に回答するための仕様体系である。

設計仮説（v3）

目的に対して最適な知能の配置を設計する。その配置において人間が含まれるかどうか、どの役割を担うかは、設計の結果であり前提ではない。

定義

Corporate OSとは何か

Corporate OSは、超人的知能が存在する世界における組織の全体アーキテクチャを設計する仕様体系である。

コンピュータのオペレーティングシステムだけを指さない。組織が何のために存在し、どの知能がどの役割を担い、判断がどう行われ、価値がどう分配され、システム自体がどう学習・進化するかを規定する。

この仕様の目的は、「自動的に正しい設計」を与えることではない。目的は、一次的な問い漏れを取り除き、残るリスクを「解の質」と「実装の質」の問題へ押し込むことにある。

仕様の構造

3層モデル

本仕様は、3つの層で構成される。抽象度の異なる3つのレンズで、同じ設計対象を照射する。

第1層

10レイヤー — 完全性の地図

AI-native組織を抽象的に捉えるための座標系。「設計すべき論点がどの種類の制度的現実に属するか」を整理する。どのレイヤーにも属さない一次論点を残してはならない。

第2層

13次元 — 具体的な設計論点

実際に「何を決めなければならないか」を問う。各次元は独立した設計課題であり、同時に因果的に接続されている。

第3層

5要素 — 各論点の必須回答形式

すべての次元に対して、Invariant / Allocation / Protocol / Exception / Metric の5つを定義する。これにより、設計の抜け漏れを形式的に検証できる。

因果接続

4つの因果チェーン

13次元は独立ではなく、因果的に接続されている。いずれかの鎖が切れると、制度全体が機能不全に陥る。

Chain A
D1 Objective → D4 Decision Rights → D11 Actuation → D5 Incentives
目的が権限を制約し、権限が実行を制約し、実行結果がインセンティブに接続される。
Chain B
D3 Membership → D4 Decision Rights → D8 Operating Model → D7 Human Dev → D2 Legitimacy
権利が権限を形づくり、権限が日常の仕事を形づくり、仕事が人間の主体性を保全してはじめて正当性が維持される。
Chain C
D9 World Model → D10 Deliberation → D4 Decision → D11 Actuation → D12 Observability → D13 Meta-Gov
状態を理解し、推論し、判断し、実行し、観測し、OS自体を更新する。制度知能の全ループ。
Chain D
D2 Legitimacy → D1 Objective → D5 Political Economy → D6 Social Order → D2 Legitimacy
正当性は入力であり出力でもある。分配と社会秩序を通じて循環する。
人間の不可譲渡な権利

4つの権利

能力ベースの「人間にしかできないから人間がやる」は設計原理にならない。AIの能力がその領域で人間を超えた時点で崩壊する。代わりに、人間が人間であるという存在論的事実に基づく4つの権利を定義する。

離脱権
組織・システムからいつでも離脱できる。AIが「全体最適が損なわれる」と主張しても妨げられない。
停止権
OSの任意の部分を即座に停止できる。事前承認は不要、事後検証は必須。
受益権
組織が生む価値の一定割合を受け取る。AIの貢献度が100%であっても維持される。
知る権利
OSの動作状態、意思決定の根拠、自分に関するデータの利用状況を理解可能な形式で知ることができる。
通時的設計

3フェーズ移行モデル

設計の目標状態はPhase A（AI運営、人間参加）。現在のAI能力ではPhase Aを即座に実現できないため、Phase H → C → Aの移行経路を設計する。

Phase H
Human Sovereignty
人間が最終権限を保持。AI運営の基盤を構築する出発点。
Phase C
Co-Sovereignty
領域別にAI運営を拡大。人間の参加モデルを検証する中間段階。
Phase A — 目標状態
Algorithmic Sovereignty
AIが組織を運営し、人間が参加者として権利を行使する。
メタガバナンス

ハードコア / ソフトシェル

OSが自己改定できなければ硬直化し、無制限に改定可能であれば暴走する。この均衡を2層構造で設計する。

ハードコア（改定不可能）：4つの不可譲渡な権利、およびハードコア自体の改定禁止。いかなる手続きを経ても変更できない。

ソフトシェル（改定可能）：権限配分の比率、手続き、評価基準、閾値、ワークフロー。定められた改定手続きに従って変更できる。改定権は提案権・審査権・承認権・拒否権の4つに分権される。

設計原則

8つの設計原則

01
AI運営を設計の出発点とする。「人間がAIにどこまで任せるか」ではなく、「AIが運営する組織に人間がどう参加するか」を問う。
02
上位レイヤーなき下位最適化を禁止する。目的・権利・分配が定義されない状態で認知・実行を最適化すると、正当性のないシステムになる。
03
人間の参加は設計対象そのものである。「残余タスク処理者」でも「最終承認者」でもなく、主体性・意味・成長・関係性の設計が必要。
04
能力ベースの正当化を設計原理にしない。「人間にしかできないから」はAI能力が超えた時点で崩壊する。権利は受益者性と手続的正統性に基づく。
05
権利と職能を区別する。不可譲渡なのは「権利」であり「職能」ではない。職能は委譲しうるが、権利はAI能力と無関係に維持される。
06
反省は出力改善だけでなく制度改善へ拡張する。ポリシー、権限配分、文化設計の修正まで含む。
07
分配の設計を後回しにしない。知能の限界より分配のルールがボトルネックになる時代が来る。
08
速度より正当性、正当性より更新可能性。短期は速度、中期は正当性、長期は更新可能性が決定的になる。
