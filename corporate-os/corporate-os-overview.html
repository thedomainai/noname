Overview Document

Corporate OS

AI-Native Institution / Operating System Design Framework
April 2026
中核の問い

人間より優れた知能が存在する世界において、組織はどのように運営され、駆動されるべきか。 Corporate OS は、この問いに対して構造的に回答するための仕様体系である。

設計仮説

目的に対して最適な知能の配置を設計する。その配置において人間が含まれるかどうか、どの役割を担うかは、設計の結果であり前提ではない。

Definition

Corporate OSとは何か

制度のための OS
Corporate OS は、単なる agent runtime や workflow tool ではない。組織が何のために存在し、誰が権利を持ち、どう判断し、どう実行し、どう学習し、どう更新するかまで含んだ制度設計の仕様である。

さらに、知識を会話や文書の断片のまま放置せず、再利用可能な skill を第一級の外部知識資産として扱うことで、制度知能を episodic ではなく cumulative にする。
01 / WHY
存在理由と正当性
目的、境界、権利、責任を決める。
02 / HOW
日常運用の構造
役割、ループ、例外、fallback を設計する。
03 / THINK
状態・記憶・skill
世界モデル、memory、skill asset を制度的に扱う。
04 / EVOLVE
観測と改定
安全に観測し、 review を経て OS 自体を更新する。

Knowledge Grammar

workflow / memory / skill の区別

今回の拡張で最も重要なのは、知識を単に保存するのではなく、制度的に再利用できる単位へ変換する点にある。

OPERATING FLOW
workflow
制度運用の流れ。誰が起点となり、どこで handoff し、どこに exception と fallback があるかを定義する。
→
INSTITUTIONAL MEMORY
memory
制度が保持する知識と履歴。semantic, episodic, procedural, evaluative, normative に分かれる。
→
EXECUTABLE REUSE
skill
memory を executable にし、 workflow に再注入する reusable capability unit。発見・評価・再利用・改定の対象になる。
ポイント: playbook やコードが存在するだけでは制度資産にならない。discoverability、relation、approval state、quality gate を持つとき、はじめて cumulative な知能資産になる。
Causal Engine

4つの因果チェーン

13次元は独立ではない。とくに D9-D13 は、制度知能が「知る -> 判断する -> 実行する -> 観測する -> 更新する」までを閉じる中核ループである。

Chain A
D1 Objective → D4 Decision Rights → D11 Actuation → D5 Incentives

Chain B
D3 Membership → D4 Decision Rights → D8 Operating Model → D7 Human Dev → D2 Legitimacy
権利が日常の運用と人間の意味を通じて、長期的な正当性へ戻る。

D9 World Model & Skill Assets → D10 Deliberation → D4 Decision → D11 Actuation → D12 Observability → D13 Meta-Gov
状態と skill 資産を理解し、提案を形成し、実行し、観測し、OS と skill 資産自体を更新する。制度知能の主ループ。

D2 Legitimacy → D1 Objective → D5 Political Economy → D6 Social Order → D2 Legitimacy
正当性は入力であり出力でもある。分配と社会秩序を通じて循環する。

Human Guarantees

4つの不可譲渡な権利

能力ベースの「人間にしかできないから人間がやる」は設計原理にならない。代わりに、人間が人間であることに基づく rights bundle を定義する。

rights bundle の考え方: 4つの権利は別々の福利厚生ではない。理解し、停止し、価値を受け取り、最後は離脱できるという一連の最低保証として機能する。
離脱権
組織・システムからいつでも離脱できる。全体最適の名目でも妨げられない。
停止権
OS の任意の部分を即座に停止できる。事前承認不要、事後検証必須。
受益権
組織が生む価値の一定割合を受け取る。AI 貢献度と無関係に保持される。
知る権利
動作状態、判断根拠、自分に関するデータ利用を理解可能な形式で知ることができる。
01
知る
状況と影響を理解できるから、形式的承認者で終わらない。
02
止める
価値違反や異常時に、通常フローを迂回して介入できる。
03
受け取る
制度が生む価値に対し、人間が受益主体であり続ける。
04
離脱する
最終的に制度から退出できることで、従属ではなく参加になる。

Principles

8つの設計原則

8つの原則は同じ階層に並んでいるわけではない。読む順序としては、前提 -> ガバナンス -> 人間参加 -> 進化 の4群に分けると理解しやすい。

A. 出発点の原則
what frame we start from
01
AI運営を設計の出発点とする。 「人間がAIにどこまで任せるか」ではなく、「AIが運営する組織に人間がどう参加するか」を問う。
02
上位レイヤーなき下位最適化を禁止する。 目的・権利・分配が定義されない状態で認知・実行を最適化すると、正当性のないシステムになる。
B. ガバナンスの原則
how legitimacy survives capability growth
04
能力ベースの正当化を設計原理にしない。 「人間にしかできないから」は AI 能力が超えた時点で崩壊する。権利は受益者性と手続的正統性に基づく。
05
権利と職能を区別する。 不可譲渡なのは「権利」であり「職能」ではない。職能は委譲しうるが、権利は AI 能力と無関係に維持される。
C. 人間参加の原則
how humans remain participants, not clerks
03
人間の参加は設計対象そのものである。 「残余タスク処理者」でも「最終承認者」でもなく、主体性・意味・成長・関係性の設計が必要。
07
分配の設計を後回しにしない。 知能の限界より分配のルールがボトルネックになる時代が来る。
D. 進化の原則
how the OS keeps changing without collapse
06
反省は出力改善だけでなく制度改善へ拡張する。 ポリシー、権限配分、文化設計、skill 資産の更新まで含む。
08
速度より正当性、正当性より更新可能性。 短期は速度、中期は正当性、長期は更新可能性が決定的になる。
