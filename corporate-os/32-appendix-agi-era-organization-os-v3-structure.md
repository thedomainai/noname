# AGI/ASI時代の組織OS設計 v3 — 構造設計書

## このドキュメントの目的

既存ドキュメント（v1: 第1部〜第3部、v2: 第4部）をゼロベースで再構築するための構造設計を定める。ここで定めるのは各セクションの「何を扱い、なぜ必要か」のみであり、各セクションの中身は別途執筆する。

## ゴール（不変）

> **AGI/ASI時代において、組織がAIを主要な知的主体として安全・持続可能・正当に運用できる状態を設計・実現する。**

## 既存構造の問題診断

### v1（第1部〜第3部）の構造

v1は3つの問いで構成されている。

1. 設計空間の全体像はなにか（第1部: 10レイヤー）
2. 今どこにいて、どこを目指すべきか（第2部: 5レベル成熟度）
3. 目標地点をどう構築するか（第3部: Level 4実装設計）

### 構造的問題

**問題1: ゴールとのカバレッジ不整合**

ゴールは「AGI/ASI時代」全体を射程に含むが、第3部はLevel 4（AGI初期、Phase H相当）の設計しかカバーしていない。Phase C（Co-Sovereignty）とPhase A（Algorithmic Sovereignty）の設計が構造上存在しない。3つの問いのうち3番目（「どう構築するか」）がゴールの一部しか答えていない。

**問題2: 第4部の構造的不整合**

v2の第4部は7つの大論点を論じ、Phase H/C/A、4つの不可譲渡な権利、メタガバナンスの2層構造など重要な概念を導入した。しかし、これらはv1の3部構造に上書きする形で追加されており、新旧の概念が混在している。例えば「人間の6職能」はv1第3部で定義され、v2第4部で「4権利+3カテゴリ職能」に再定義されるが、両方が文書内に併存したままである。

**問題3: フレームの多重化**

「10レイヤー」「5レベル」「Level 4の上位構造（A〜E）」「7大論点」という4つの異なる整理フレームが並列しており、読者が全体像を掴みにくい。設計空間の地図（10レイヤー）と実装構造（A〜E）と通時的設計（Phase H/C/A）の関係が明示されていない。

**問題4: 通時性の構造的欠如**

AI能力の変化に応じて設計が変わるという洞察はv2で提示されたが、v1の構造がLevel 4に固定されているため、通時的な移行設計を配置する場所が構造上ない。

## v3が答えるべき問いの体系

ゴールを分解すると、以下の8つの問いに答える必要がある。

| # | 問い | ゴールとの対応 |
|---|---|---|
| Q1 | この設計は何を扱い、何を扱わないのか | 設計空間の定義 |
| Q2 | 何のために、誰のために存在するのか | 「正当に」の基盤 |
| Q3 | 誰が何を決め、誰が責任を負うのか | 「安全・正当に」の統治設計 |
| Q4 | AI能力の変化にどう適応するか | 「AGI/ASI時代において」の通時性 |
| Q5 | 日々どう運用するのか | 「運用できる状態」の具体設計 |
| Q6 | 安全性と信頼をどう確保するか | 「安全に」の技術・制度設計 |
| Q7 | 人間はどう存在するか | 「持続可能に」の人間側設計 |
| Q8 | 複数の組織OSが共存する世界をどう設計するか | 社会レベルの「正当に」 |

v1+v2ではQ4とQ8に構造的に答えていない。Q2、Q3はv2で概念レベルの答えがあるが、v1の構造に統合されていない。

## 構造設計の検討

### 検討した構造案

**案A: 10レイヤー×3フェーズのマトリクス**

30セル（10レイヤー × Phase H/C/A）を網羅的に記述する構造。各レイヤーが各フェーズでどう変化するかを明示できる。しかし、30セルは冗長であり、レイヤー間の依存関係が見えにくく、読者が各セルを読んでも全体設計の意図が掴めない。設計空間の「地図」としては有効だが、ドキュメントの「構造」としては不適切である。

**案B: 時間軸中心（Phase H→C→Aを部にする）**

Phase H、Phase C、Phase Aの順に設計を記述する構造。移行ストーリーが明確で「今どこにいて次にどうする」が直感的に分かる。しかし、各フェーズ内で同じ設計次元（統治・運用・安全等）を繰り返す冗長さがあり、フェーズ横断の共通原理が散在する。Phase Aの具体設計は現時点で困難であるため、Phase Aの章が薄くなり構造のバランスが崩れる。

**案C: 問い中心（Q1〜Q8を部にする）**

上記8つの問いをそのまま部にする構造。カバレッジの検証が容易であり、論点の漏れがない。しかし、問い間の相互依存が多く前後参照が頻発する。また、実装設計との距離が遠く、「この構造に従えば実装できる」感覚が弱い。

**案D: 原理→統治→運用→適応の4部構成（採用案）**

ドキュメントの読者が辿るべき思考の順序に沿った構造。「まず全体像と原理を理解し、次に統治構造を設計し、次に運用を設計し、最後に時間軸上の変化と外部接続を設計する」という流れ。Q1〜Q8が構造的にカバーされ、通時性は第4部に集約しつつ第2部・第3部でもフェーズ依存性をサブセクションで扱える。

### 案Dを採用する理由

1. **ゴールの全要素がカバーされる**: 「安全」→第2部+第3部、「持続可能」→第1部+第3部、「正当」→第2部+第4部、「運用できる状態」→第3部、「AGI/ASI時代」→第4部
2. **読者の認知負荷が低い**: 「原理→統治→運用→適応」は設計者の自然な思考順序に沿う
3. **通時性が構造的に組み込まれる**: 第4部で移行設計を集約しつつ、各部でフェーズ依存性を扱える
4. **10レイヤーが「地図」として機能する**: 第1部で設計空間の地図として定義し、各部のセクションが対応レイヤーを明示することで漏れを検証可能にする。ただし10レイヤーはドキュメントの「構造」ではなく「チェックリスト」として使う
5. **既存コンテンツの再利用性が高い**: v1第1部→第1部、v1第3部→第3部、v2→第2部+第4部に自然に対応する

### 10レイヤーの扱いに関する設計判断

v1の10レイヤー（L1〜L10）は設計空間の「次元」の定義として有効である。しかし、v1ではドキュメントの構造と設計空間の構造が混同されていた。v3では以下のように分離する。

- **10レイヤーはドキュメントの章立てには使わない**: 10レイヤーをそのまま章にすると、統治・運用・適応が各レイヤーに分散し、設計の意図が見えなくなる
- **10レイヤーは設計空間のチェックリストとして使う**: 第1部で定義し、各セクションが「このセクションはL1〜L3を扱う」と明示する。ドキュメント末尾のカバレッジ検証マトリクスで、10レイヤー×セクションの対応を示し、漏れがないことを保証する
- **レイヤーの再分類を検討する**: v1の10レイヤーがv3の構造でも最適かどうかを評価する（後述の「10レイヤーの再評価」を参照）

## v3のドキュメント構造

### 全体構成

```
序章    ゴールとこのドキュメントの読み方
第1部   原理と設計空間 — 何を設計し、何を原則とするか
第2部   統治設計 — 誰が何を決め、どう責任を負うか
第3部   運用設計 — 日々どう動かし、どう安全を確保するか
第4部   移行と適応の設計 — AI能力の変化と外部環境にどう対応するか
付録    カバレッジ検証マトリクス / 用語定義 / 既存設計との差分
```

### 序章: ゴールとこのドキュメントの読み方

**扱う内容**: ドキュメントのゴール、4部構成の論理的必然性、読み方のガイド。

**なぜ必要か**: 読者が「この構造に従えば、ゴールに到達するために必要な全ての設計判断が網羅される」と確信するための導入。各部がゴールのどの要素に対応するかを明示し、部間の依存関係を示す。

**対応する問い**: 全体の構造を示すメタ問い。

**v1からの変更点**: v1の第0部は「3つの部がゴールを達成する理由」を説明していたが、v3ではゴールの分解（8つの問い）を明示し、各部がどの問いに答えるかのマッピングを提供する。4部構成の選定理由と、検討した他の構造案（マトリクス型、時間軸型）を棄却した根拠も含める。

### 第1部: 原理と設計空間

#### §1.1 組織OSの定義と射程

**扱う内容**: 「AGI/ASI時代の組織OS」が何を意味するか。OSは認知アーキテクチャではなく、目的・権力・分配・文化・人間・技術・進化を包含する多層的秩序であること。何を扱い、何を扱わないか（射程の限定）。

**なぜ必要か**: 設計対象の定義なしに設計はできない。既存のAI組織論が認知アーキテクチャに縮約されやすい誤りを防ぐ。

**対応する問い**: Q1（この設計は何を扱い、何を扱わないのか）

**対応レイヤー**: 全レイヤーの包括的定義

**既存コンテンツ**: v1第1部「中核認識」「OSの定義」「既存議論が陥りやすい誤り」を再構成

#### §1.2 設計空間 — 10の設計次元

**扱う内容**: 組織OSが扱う10の次元（レイヤー）の定義と相互依存関係。各次元が欠落した場合の失敗パターン。3つの大領域（組織は何者か / どう生きるか / どう知り・行動し・進化するか）。

**なぜ必要か**: 後続の全てのセクションが「どの次元を扱っているか」を参照するための地図。設計の漏れを検証する基準。

**対応する問い**: Q1（設計空間の定義）

**対応レイヤー**: L1〜L10の全定義

**既存コンテンツ**: v1第1部「全体俯瞰表」「各レイヤーの定義」「3つの大領域」「レイヤー間の依存関係」をそのまま再利用。ただし「10レイヤーの再評価」（後述）の結果に基づき調整する可能性あり。

**設計判断: 10レイヤーの再評価**

v1の10レイヤーはv3でも有効かどうかをゼロベースで評価する。

維持すべき理由:
- 10レイヤーは「OSが扱うべき次元の地図」として機能しており、v2の7大論点でも「10レイヤーの設計空間は変更不要」と結論づけている
- レイヤー間の依存関係（上位→下位の拘束、中位の翻訳、下位の実現、L10の改定可能性）は有効な構造的洞察である

調整が必要な可能性:
- L4（社会・文化・信頼）とL5（人間の主体性・成長・意味）は、v2で導入された「不可譲渡な4権利」「意味の構造的保証」と重複・緊張関係にある。統合が必要かもしれない
- L10（メタガバナンス）は、v2で導入された「ハードコア/ソフトシェルの2層構造」と統合すべきである
- レイヤーの「番号付け」が上位→下位の印象を与えるが、v3の構造では「次元」として並列に扱う方が適切かもしれない

この再評価は本構造設計書では結論を出さず、第1部の執筆時に確定する。

#### §1.3 設計原則

**扱う内容**: OS設計全体を貫く原則。v1の5原則とv1第3部の6原理を統合・再定義する。Phase H/C/Aに依存しない、全フェーズ共通の原則のみをここに置く。

**なぜ必要か**: 第2部以降の設計判断が準拠すべき基準。原則がなければ、設計判断の一貫性を保てない。

**対応する問い**: Q2（何のために存在するのか）の基盤部分

**対応レイヤー**: 全レイヤーに横断的

**既存コンテンツ**: v1第1部「OSを設計するときの5つの原則」+ v1第3部「6つの原理」を統合。重複を排除し、v2の知見（能力ベースの正当化の棄却、権利と職能の区別等）を反映した原則体系に再編する。

**統合の方針（候補）**:
- v1の5原則（上位レイヤーなき下位最適化の禁止 / 人間は設計対象 / 反省の制度改善への拡張 / 分配設計の先行 / 速度より正当性より更新可能性）は概念レベルの指針であり維持
- v1第3部の6原理（Purpose first / Governance by design / AI as default operator / Human as constitutional authority / Selective human intervention / Agency preservation as KPI）はLevel 4実装に寄りすぎている。フェーズ非依存の形に抽象化する
- v2の「能力ベースの正当化を棄却する」「権利と職能を区別する」は原則に昇格させるべき

#### §1.4 能力の時代変化と設計への含意

**扱う内容**: AI能力がAGI→ASIへと変化することで、設計の前提がどう変わるかの概観。「能力が低い時代は下位レイヤーがボトルネック、能力が高い時代は上位レイヤーがボトルネック」という洞察を構造化する。Phase H / C / Aの定義と概要（詳細は第4部）。

**なぜ必要か**: 第2部・第3部の設計が「どのフェーズでの設計か」を常に意識するための前提共有。通時性を全体に浸透させるための布石。

**対応する問い**: Q4（AI能力の変化にどう適応するか）の前提部分

**対応レイヤー**: 全レイヤーに横断的

**既存コンテンツ**: v1第1部「AGIとASIで重要になるレイヤーの変化」+ v2第4部「Phase H/C/A」の定義部分。v1第2部の5レベル成熟度は「診断ツール」としての価値はあるが、v3の構造では成熟度モデルをドキュメントの章にはせず、ここで「現在地の評価フレーム」として簡潔に参照する。

### 第2部: 統治設計

#### §2.1 権威の原理 — 誰が・なぜ決定権を持つか

**扱う内容**: 人間の権威の根拠を「能力」から切り離し、「受益者性」「手続的正統性」「能力」の3原理の複合として定義する。3原理の重み付けがPhase H→C→Aで変化することを示す。権威の「根拠」と「行使方法」の分離。

**なぜ必要か**: 統治設計の全ての判断は「誰が決定権を持つか」に依存する。この根拠が明確でなければ、権限配分・メタガバナンス・人間の権利の設計が恣意的になる。v1では「Human as constitutional authority」が所与とされていたが、v2でその根拠の脆弱性が指摘された。ここで根拠を確立する。

**対応する問い**: Q2（何のために、誰のために存在するのか）、Q3（誰が何を決め、誰が責任を負うのか）

**対応レイヤー**: L1（存在目的・外部正当性）、L2（憲法・権力・責任）

**既存コンテンツ**: v2第4部の大論点1（小論点1.1〜1.4の全て）を再構成

#### §2.2 人間の不可譲渡な権利

**扱う内容**: 4つの不可譲渡な権利（離脱権・停止権・受益権・知る権利）の定義、根拠、行使条件。権利と職能の区別。職能を「不可譲渡な権利」「条件付き職能」「委譲可能な職能」の3カテゴリに再編した体系。

**なぜ必要か**: 全てのAI能力が人間を超えた状態でも維持される「底」を定義する。これがないと、Phase Aの設計で人間が完全に排除されうる。また、v1の「人間の6職能」がASI段階で半数以上崩壊することがv2で示されたため、能力ベースの職能定義から権利ベースの定義への転換が必要。

**対応する問い**: Q2（誰のために存在するのか）、Q7（人間はどう存在するか）

**対応レイヤー**: L2（憲法・権力・責任）、L5（人間の主体性・成長・意味）

**既存コンテンツ**: v2第4部の大論点4（小論点4.1〜4.4）+ v1第3部「人間の6職能」の再定義

#### §2.3 統治構造 — 権限配分のアーキテクチャ

**扱う内容**: 組織OSの統治構造の設計。Purpose Council / Governance Kernel / Domain Cells / Execution Fabric / Reflection & Evolution Engineの5層構造（またはその再設計版）。各層の役割、権限、人間とAIの分担。リスク分類、Trust Score、Composite Riskの設計。

**なぜ必要か**: 権威の原理（§2.1）と権利（§2.2）を、実際の統治構造に落とし込む。「原理はあるが構造がない」状態を防ぐ。

**対応する問い**: Q3（誰が何を決め、誰が責任を負うのか）

**対応レイヤー**: L1（存在目的）、L2（憲法・権力・責任）、L3（経済・分配）

**既存コンテンツ**: v1第3部「OSの上位構造（A〜E）」を再構成。v2の知見（動的権限配分、3原理の重み付け）を統合

**フェーズ依存性**: Phase H / C / Aで統治構造がどう変化するかのサブセクションを含める。ただし移行のトリガーと手順は第4部で扱う。

#### §2.4 メタガバナンス — OS自体の改定権の設計

**扱う内容**: ハードコア（改定不可能な層）とソフトシェル（改定可能な層）の2層構造。改定の提案権・審査権・承認権・拒否権の分権設計。改定速度の3段階設計。

**なぜ必要か**: OSが自己改定できなければ硬直化し、無制限に改定可能であれば暴走する。この両極の間で均衡点を設計するのがメタガバナンスであり、AGI/ASI時代のOS設計において最も本質的な設計対象の一つである。

**対応する問い**: Q3（誰が責任を負うのか）のメタレベル

**対応レイヤー**: L10（学習・適応・メタガバナンス）

**既存コンテンツ**: v2第4部の大論点3（小論点3.1〜3.4の全て）を再構成

### 第3部: 運用設計

#### §3.1 業務ループ — Sense→Evolveの8ステップ

**扱う内容**: 業務ループ（Sense → Understand → Deliberate → Gate → Act → Observe → Reflect → Evolve）の設計。各ステップの詳細と、人間の関与ポイント。Constitution Load（Step 0）の設計。

**なぜ必要か**: 第2部の統治設計を日々の業務に翻訳する具体的なプロセス設計。統治原理がどれほど優れていても、運用プロセスがなければ動かない。

**対応する問い**: Q5（日々どう運用するのか）

**対応レイヤー**: L6（仕事・役割・運用モデル）、L7（認知・知識・意思決定）

**既存コンテンツ**: v1第3部「業務設計」「データフロー/業務フロー（Step 0-8）」をほぼそのまま再利用

#### §3.2 自律度制御 — Autonomy Modeとリスク分類

**扱う内容**: Autonomy Mode（A0〜A5）の定義と適用基準。Composite Riskの算出方法。業務分類（Type 1〜5）とAutonomy Modeの対応。

**なぜ必要か**: 「何をどこまで自律させるか」は運用設計の最も重要な判断であり、安全性の中核。リスクに応じた自律度の段階設計がなければ、「全て手動」か「全て自律」の両極端に陥る。

## v1第2部（5レベル成熟度）の扱いに関する設計判断

v1第2部の5レベル成熟度モデル（Level 1〜5）は、v3の主要構造には組み込まない。理由は以下の通り。

1. **v3の構造はフェーズ（H/C/A）を通時軸にしている**: 5レベル成熟度とPhase H/C/Aの両方を通時軸として保持すると、読者に2つの異なるスケールを同時に要求することになり、認知負荷が過大になる
2. **5レベルは「組織の現在地の診断」に有効だが、「設計の構造」ではない**: 「あなたの組織はLevel 2です」という診断は有用だが、設計ドキュメントの章立てとしてはPhase H/C/Aの方が適切である
3. **Level 4の実装設計がv1第3部に対応していた問題を解消する**: v1ではLevel 4のみが実装設計を持ち、Level 5は「射程として定義」に留まっていた。v3ではPhase H/C/Aの全てに設計が配置される

ただし、5レベル成熟度モデル自体は「現在地の診断フレーム」として§1.4で簡潔に参照する。Level 1〜3はPhase Hの前段階、Level 4はPhase H、Level 4.5はPhase C移行期、Level 5/6はPhase C/Aに対応する関係を示す。
