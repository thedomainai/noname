02 — Problem

解決する課題

DDプロジェクトでは、成果物（Q&A・ドキュメント・チェックリスト・ファインディング）がメール・VDR・スプレッドシート等に分散し、「誰が何を知っているか」「どこまで完了したか」「何と何が関連するか」が見えない。

構造的な問題

これらの情報がサイロ化しているため、以下が不可能：

横断検索

「この質問に関連するドキュメントはどれか」が追跡できない

進捗可視化

「全領域で何%完了したか」が見えない

関連性追跡

「このファインディングはどの質問から発見されたか」が辿れない

重複検知

「似た質問が過去にあったか」「このチェック項目は誰かが既に確認したか」がわからない

結果として発生すること

Q&Aの重複・漏れ

複数領域が独立して質問を作成し、対象会社に同じ質問が複数回届く。信頼関係の毀損と回答遅延を引き起こす。毎朝の手動消込では限界がある。

進捗の不可視

DD全体の進捗がチェックリストの消化率でしか測れない。「どの領域が遅れているか」「ブロッカーは何か」がリアルタイムで把握できず、週次会議まで問題が潜伏する。

発見事項の散逸

各領域のDD担当が発見したリスク・機会が、個別のレポートに分散する。領域横断的なリスク（例: IT負債が事業計画に与える影響）が統合されない。

VDRのUI障壁

既存VDRは機能過多でUIが複雑。必要な資料への到達に時間がかかり、アドバイザーの生産性を下げている。AI対応のVDRプレーヤーが不在。

03 — Deep Dive

課題の構造的分析

表面的な「管理が追いつかない」の背後には、DD実行の構造に起因する7つの問題が存在します。これらを正しく捉えなければ、単なるタスク管理ツールに終わってしまいます。

ISSUE 01
フェーズ遷移の判断基準が暗黙知

ソーシング→NDA→IM→LOI→DD→SPA→クロージングの各フェーズ遷移条件が、統括責任者の頭の中にしかない。「LOIを出してよいか」の判断に必要な情報（初期バリュエーション・戦略適合度・リスクフラグ）が構造化されていないため、毎回ゼロベースで議論が発生します。

ISSUE 02
Q&A重複検出の計算量爆発

DD期間中のQ&Aは増加し続けます。新しい質問を追加するたびに、既存の全質問との類似度を人手で確認する必要があり、O(n²)のコストが発生します。pg_trgmによる類似度検索で計算量をO(n)に抑えることが可能ですが、既存ツールにはこの機能がありません。

ISSUE 03
チェックリストの案件別カスタマイズ

BDD/FDD/LDD/ITDD/HRDDのチェックリストテンプレートは存在するが、案件ごとに必要な項目が異なります（例: IT企業のDDではITDDの比重が大きく、製造業ではオペレーションDDが追加される）。テンプレートの適用と案件別カスタマイズの両立が必要です。

ISSUE 04
事業計画の整合性検証が属人的

Pre-DDで作成する事業計画には3パターン（スタンドアローン/コスト改善/トップライン成長）があり、DD実行中に発見された事実との整合性を継続的にチェックする必要があります。この検証は経験豊富なアナリストに依存しており、属人化しています。

ISSUE 05
発見事項の重要度判定と伝播

FDDで発見された「簿外債務」がLDDの契約条項やSPAの表明保証にどう影響するかを、領域横断的に追跡する仕組みがありません。Critical/High/Medium/Lowの重要度分類はあるが、他領域への波及効果が見えません。

ISSUE 06
ドキュメントのバージョン管理

IM（情報メモランダム）・財務諸表・契約書等の資料は、DD期間中に複数回更新されます。最新版がどれか、誰がいつ更新したか、前版との差分は何かを追跡する手段が不十分です。

ISSUE 07
ロングリストからDDへの連続性

ロングリストで蓄積した企業情報（フィットスコア・接触履歴・公開情報）がDD開始時に引き継がれません。Pre-DDの仮説構築に使えるはずの情報が、ツール間の断絶により活用されていません。

失敗コスト: Q&Aの重複で対象会社の信頼を失う。チェックリストの漏れでリスクを見逃す。発見事項が統合されず、投資判断の精度が下がる。DD期間の延長でアドバイザー費用が膨張する。
04 — Solution

05 — Application Structure

アプリケーション構成

DD Navigatorは2階層の情報構造で設計されています。トップレベルでは横断管理機能を提供し、案件詳細では個別案件に紐づく情報を統合管理します。

トップレベル機能

サイドバーから直接アクセスできる、全案件を横断した管理機能です。

TOP LEVEL 01
ダッシュボード
全案件のサマリーを一画面で把握します。アクティブ案件数・未回答Q&A数・直近の更新を表示し、注意が必要な案件にドリルダウンできます。
  • アクティブ案件数と完了案件数の集計
  • 未回答Q&A数とブロッカー件数の表示
  • 直近の更新アクティビティフィード
  • 注意が必要な案件のハイライト表示
TOP LEVEL 02
案件管理（Deal Pipeline）
ソーシングからクロージングまでの9フェーズで全案件を管理します。各案件のフェーズ・ステータス・進捗率を一覧し、個別案件の詳細にアクセスできます。
  • 9フェーズのパイプラインビュー（Sourcing → Closing）
  • 案件ごとのフェーズ・ステータス管理（active / paused / completed / abandoned）
  • 案件一覧での進捗サマリー表示
  • 案件詳細へのナビゲーション
TOP LEVEL 03
ロングリスト（Longlist Bridge）
案件化前の企業情報を管理します。フィットスコア・業種・地域・企業概要を構造化データとして保存し、DD開始時に案件へ自動引き継ぎできます。
  • ロングリスト企業のCSVインポート
  • フィットスコア（1-5）による優先順位管理
  • 業種・地域・売上規模・従業員数の構造化データ
  • ロングリスト企業からのワンクリック案件作成
TOP LEVEL 04
QA管理
全案件のQ&Aを横断的に管理します。未回答Q&Aの一覧表示、期限超過アラート、領域別フィルタリングにより、Q&A全体の状況を把握できます。
  • 全案件のQ&A一覧と横断検索
  • 未回答Q&Aと期限超過のフィルタリング
  • カテゴリ別（財務 / 法務 / HR / IT / ビジネス）の集計
  • Q&A重複検出の統計レポート

案件詳細機能

案件を開いた時に利用可能な、個別案件に紐づく情報管理機能です。

DEAL MODULE 01
Q&A Hub
案件別のQ&Aを管理します。pg_trgm（PostgreSQLのトライグラム類似度検索）を使い、新しい質問を追加する際に既存質問との類似度を自動判定。重複質問の対象会社への送付を防ぎます。
  • 質問のステータス管理（下書き → 送信済 → 回答済 → 要確認）
  • カテゴリ分類（財務 / 法務 / HR / IT / ビジネス）
  • 優先度と期限の設定、担当者アサイン
  • 類似質問の自動検出と重複アラート
DEAL MODULE 02
Document Vault
DD関連資料をカテゴリ別に管理します。IM・財務諸表・契約書・技術文書等を案件に紐づけて格納し、チームメンバーが必要な資料に即座にアクセスできる環境を提供します。
  • カテゴリ別のドキュメント管理（IM / 財務 / 法務 / 技術 / HR / その他）
  • ファイルサイズ・アップロード日時・アップロード者の追跡
  • ストレージクォータ管理
  • ファイルタイプ別のアイコン表示（PDF / Word / Excel / 画像）
DEAL MODULE 03
Findings Tracker
DDで発見されたリスク・機会・課題・情報を構造化して管理します。重要度（Critical / High / Medium / Low）とステータス（Open / Under Review / Resolved / Accepted）の2軸で追跡し、投資判断の材料を整理します。
  • 4カテゴリ分類（リスク / 機会 / 課題 / 情報）
  • 4段階の重要度判定とステータス管理
  • 影響度の記述と是正策の提案
  • 重要度別のフィルタリングとソート
DEAL MODULE 04
DD Checklist
BDD / FDD / LDD / ITDD / HRDD の5種類のチェックリストテンプレートを提供します。案件ごとにテンプレートを適用し、カスタム項目の追加・削除が可能。カテゴリ別の進捗率を自動算出します。
  • 5種類のDDテンプレート（BDD / FDD / LDD / ITDD / HRDD）
  • カテゴリ別グルーピングと進捗バー
  • 完了日時・完了者の自動記録
  • テンプレートの案件別カスタマイズ

意図的に含めないもの

  • VDRの完全代替: Intralinks/Dataroomの全機能は再実装しない。資料の整理・閲覧に特化
  • バリュエーション自動算出: DCF/マルチプルの計算はスプレッドシートの領域
  • 外部アドバイザーポータル: 初期版はチーム内利用に限定。外部共有はエクスポートで対応
  • SPA条文の自動生成: 法務レビューの自動化は範囲外
  • ワークフローの自動承認: DD実行の判断は人間が行う設計を維持
06 — User Flow

使い方

DD Navigatorは、案件のライフサイクルに沿って利用されます。LOI後のDD開始から、DD完了・SPA交渉までの典型的なフローを示します。

DD開始セットアップ

1
案件作成
対象会社・業種・DD期間を登録。ロングリストから引き継ぎ可能
2
テンプレート適用
BDD/FDD/LDD等のチェックリストテンプレートを選択・カスタマイズ
3
資料アップロード
IMや対象会社提供資料をカテゴリ別に格納
4
Q&A開始
初期Q&Aリストを作成。重複チェックを有効化
5
DD実行
チェックリスト消化・Q&A管理・発見事項記録を並行実施

ロール別フロー

朝
ダッシュボードを開く → アクティブ案件数・未回答Q&A数・直近の更新を確認 → 注意が必要な案件にドリルダウン
週次
パイプラインビューで全案件のフェーズ分布を確認 → 各案件のチェックリスト消化率で遅延を検出 → ブロッカーとなっているQ&Aや未取得資料を特定
判断時
案件の発見事項一覧をCritical/Highでフィルタ → リスクの影響度と是正策を確認 → フェーズ遷移（DD→SPA交渉）の判断材料を整理
日次
新しいQ&Aを追加 → 類似質問アラートが「既存Q&A #23と82%類似」と表示 → 重複を確認し、既存質問に統合するか新規として作成するか判断
日次
Q&Aステータスフィルタで「送信済・未回答」を表示 → 期限超過のQ&Aを検出 → 対象会社へのリマインド要否を判断
都度
対象会社から回答が届く → Q&Aステータスを「回答済」に更新 → 追加確認が必要な場合は「要確認」に変更 → 各領域担当に回答を共有
DD実行
チェックリストを開く → 担当カテゴリ（BDD/FDD/LDD等）の項目を順次消化 → 完了した項目にチェックを入れると日時・担当者が自動記録
発見時
調査中にリスクを発見 → Findings Trackerに「顧客集中リスク: 売上の45%が上位3社」をCriticalで登録 → 影響度と是正策を記述 → DD統括に自動通知
資料確認
Document Vaultでカテゴリフィルタ「財務」を選択 → 最新の財務諸表を確認 → 追加資料が必要な場合はQ&Aを起票
月次
ダッシュボードでアクティブ案件数と完了案件数を確認 → パイプラインの健全性（各フェーズの滞留件数）を把握
判断時
DD完了案件の発見事項サマリーを確認 → Critical/Highの発見事項数と内容を評価 → 投資実行の意思決定に活用
