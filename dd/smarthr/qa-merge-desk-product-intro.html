QA Merge Desk

Pull Request の品質レビューを一元管理し、チェックリスト・ブロッカー・ステータス履歴を構造化する。GitHub / GitLab を横断して、QA パイプライン全体の可視性を確保するプロダクト。

01 — Target Users

誰のためのプロダクトか

事業会社の開発チームにおいて、Pull Request の品質保証プロセスを担う人々。リリース前の最後の砦として、コード品質とビジネスロジックの整合性を守る役割を持つチームが対象です。

A
QA リード / Admin
週次のパイプライン管理
  • 全 PR の QA ステータスを横断的に把握する責任
  • QA エンジニアへの作業配分を決定
  • ブロッカーの早期検出とエスカレーション判断
  • チェックリストの標準化とテンプレート管理
Q
QA エンジニア
日次のレビュー実行
  • 担当 PR のチェックリスト項目を順次消化
  • テスト結果・不具合をコメントで記録
  • ブロッカー発見時に即座に登録・通知
  • 複数 PR を同時並行でレビュー
D
開発者 / Viewer
PR 提出後のステータス確認
  • 自分の PR がどの段階にあるか確認
  • QA からのフィードバック・ブロッカーを把握
  • 修正対応後のステータス更新を待機
  • 過去の QA 結果を参照して品質基準を学習
「毎朝 GitHub と GitLab を行き来して、どの PR が今どの状態かを頭の中で組み立て直す作業から解放されたい」
--- QA リード
02 — Problem

解決する課題

PR の品質レビューは、リリースサイクルの中で最もボトルネックになりやすい工程です。レビュー状況が分散し、誰が何を見ているかが不透明になると、リリース遅延と品質劣化が同時に進行します。

5-10本
同時レビュー中の PR 数
2+
利用プラットフォーム数
30分/日
ステータス確認の工数
3-5日
平均 QA 待ち時間
ステータスの不可視

PR が pending / in_progress / testing / approved / blocked のどの段階にあるかを、GitHub のラベルと Slack の会話から推測するしかない。

チェック観点の属人化

QA で何を確認すべきかが個人の経験に依存。セキュリティチェックやアクセシビリティ検証が漏れても気づかない。

ブロッカーの埋没

テスト中に発見した阻害要因が GitHub コメントの中に埋もれる。解決済みか未解決かの判別に時間がかかる。

負荷の偏在

特定の QA エンジニアに作業が集中しても可視化されない。チーム全体のキャパシティ管理が感覚的になる。

03 — Deep Dive

課題の構造的分析

表面的な「レビューが遅い」の背後には、プロセス設計上の7つの構造的問題が存在します。これらを正しく捉えなければ、単なるタスク管理ツールに終わってしまいます。

ISSUE 01
プラットフォーム横断の情報断片化

GitHub と GitLab が混在する環境では、PR のステータスを確認するために複数のブラウザタブを行き来する必要がある。各プラットフォームのレビュー機能はプラットフォーム内で閉じており、横断的なパイプラインビューが存在しない。

ISSUE 02
QA ステータスと開発ステータスの混同

GitHub のレビュー機能（Approved / Changes Requested）はコードレビューのステータスであり、QA レビューのステータスとは別物。QA 独自の 5 段階パイプライン（pending → in_progress → testing → approved → blocked）を表現する場所がない。

ISSUE 03
チェックリストの非標準化

PR の種類（機能追加・バグ修正・セキュリティ修正・API 変更）ごとに確認すべき観点が異なるが、テンプレートが管理されていない。新メンバーは「何を確認すべきか」がわからず、経験者の暗黙知に頼る。

ISSUE 04
ブロッカーのライフサイクル管理不在

テスト中に発見した問題（外部依存の未解決・環境構築の不備・仕様の曖昧さ）がコメント欄の中に埋もれ、解決したのか・誰が対応しているのか・いつ解消されたのかを追跡する仕組みがない。

ISSUE 05
ステータス変更履歴の揮発

PR がいつ pending から in_progress に変わり、いつ blocked になったのかの履歴が残らない。「この PR は3日間 blocked のまま放置されている」という事実に誰も気づかない。

ISSUE 06
ワークロードの不均衡

QA エンジニアの担当 PR 数・ステータス別の内訳が可視化されていない。特定メンバーに testing が5本集中していても、リードが気づくのは朝会での口頭報告のタイミングまで遅延する。

ISSUE 07
老朽化 PR の検出遅延

7日以上更新されていない PR がパイプラインに滞留していても、一覧から発見するのが困難。コンフリクトが蓄積し、マージコストが時間とともに増大する。

構造的コスト: QA ボトルネックの放置は、リリース遅延だけでなく「QA を飛ばしてマージする」という品質劣化を誘発する。一度その習慣が根付くと、本番障害の発生頻度が上昇し、チーム全体の開発速度が低下する悪循環に入る。
04 — Solution

QA Merge Desk のアプローチ

GitHub / GitLab の上に乗る「QA 管制塔」。PR の品質レビューを5段階のパイプラインとして構造化し、チェックリスト・ブロッカー・コメントを一元管理します。プラットフォームを置き換えるのではなく、QA レイヤーを追加します。

01
既存ツールに被せる
GitHub / GitLab を置き換えず、Webhook で接続して QA レイヤーを上乗せ
02
5段階パイプライン
pending / in_progress / testing / approved / blocked の明確なステータス管理
03
チェックの標準化
PR 種別ごとのテンプレートで確認観点を組織知に変換
04
ブロッカーの可視化
阻害要因を独立エンティティとして管理し、解決までのライフサイクルを追跡
05
ワークロード均衡

  • QA ステータスは GitHub ラベルで運用（5段階を表現できない）
  • チェック項目は QA エンジニアの記憶頼み

  • ブロッカーは独立管理、解決/未解決を明確に
  • 担当者別ワークロードをダッシュボードで常時可視化
  • 老朽化 PR を自動検出・アラート表示
05 — Features

機能の詳細

6つのコア機能で構成されます。各機能は独立して価値を提供しつつ、組み合わさることで QA プロセス全体を支える設計です。

Feature 01
Dashboard
QA パイプライン全体を1画面で把握するダッシュボード。ステータス別 PR 数、QA エンジニア別ワークロード、老朽化 PR アラートを集約します。
  • ステータス別 PR サマリーカード（Total / Pending / In Progress / Testing / Approved / Blocked）
  • QA エンジニア別のアサイン数・ステータス内訳を棒グラフで表示
  • 7日以上更新なしの PR を自動検出してアラート一覧に表示
  • リアルタイム更新（ページ遷移なしでデータ反映）
Feature 02
PR Management
Pull Request の一覧管理とステータス遷移。GitHub / GitLab から Webhook 経由で自動登録され、QA 独自の5段階パイプラインでステータスを管理します。
  • ステータス・リポジトリ・担当者でフィルタリング可能な PR 一覧
  • PR 詳細画面で Description / チェックリスト / コメント / ブロッカーを統合表示
  • QA エンジニアのアサイン機能（ドロップダウン選択）
  • ステータス変更履歴を自動記録（誰が・いつ・何に変更したか）
Feature 03
Checklist
PR ごとの QA チェックリストを管理。テンプレートから一括適用し、完了 / 未完了をトラッキングします。
  • テンプレート一覧から選択して PR にチェック項目を一括追加
  • チェック項目の完了 / 未完了をトグルで切替（完了者・完了日時を自動記録）
  • PR 固有のチェック項目を手動追加可能
  • 3種のテンプレート: 標準 QA / API 変更 / セキュリティ修正
Feature 04
Comments
PR ごとのコメントスレッド。テスト結果の記録、QA エンジニア間の情報共有、開発者へのフィードバックを構造化します。
  • PR 詳細画面内のタイムライン形式コメント欄
  • 投稿者名・投稿日時の自動記録
  • Markdown 形式のテキスト入力に対応
  • QA 承認理由・差し戻し理由の記録に使用
Feature 05
Blockers
QA プロセスを阻害する要因を独立エンティティとして管理。発見から解決までのライフサイクルを追跡します。
  • ブロッカーの登録（タイトル・詳細説明・作成者を記録）
  • 未解決 / 解決済みのステータス管理
  • 解決時に解決者・解決日時を自動記録
  • 未解決ブロッカーがある PR は自動的に Blocked ステータスに
Feature 06
Webhook Integration
GitHub / GitLab の Webhook を受信して PR を自動登録。マルチプラットフォーム環境でも手動入力なしで QA パイプラインに流入します。
  • GitHub / GitLab の PR イベントを Webhook で受信
  • リポジトリ単位で Webhook エンドポイントを設定
  • Secret Key によるリクエスト検証
  • Webhook の有効 / 無効切替機能

意図的に含めないもの

  • コードレビュー機能: コードレビューは GitHub / GitLab のネイティブ機能を使用。QA Merge Desk は QA プロセスに特化
  • CI/CD 連携: ビルド・テスト実行は既存の CI/CD パイプラインの責務。ステータス同期は将来検討
  • 自動マージ: QA Approved 後のマージ操作は開発者が GitHub / GitLab 上で実行
  • コード解析: 静的解析・カバレッジ計測は専用ツール（SonarQube 等）の領域
06 — User Flow

使い方

QA Merge Desk は、PR のライフサイクルに沿った日次・週次の2つの業務サイクルで利用されます。初期セットアップから日常運用までの典型的なフローを示します。

初期セットアップ

1
Webhook 設定
GitHub / GitLab のリポジトリに Webhook エンドポイントを登録
2
チーム登録
QA エンジニア・Admin・Viewer のロールでメンバーを招待
3
テンプレート定義
PR 種別ごとの QA チェックリストテンプレートを作成
4
PR 自動取込
Webhook 経由で既存 PR が Pending ステータスで自動登録
5
運用開始
QA リードがアサインを行い、レビューサイクル開始

業務サイクル別フロー

QA Lead
ダッシュボードを開く → ステータス別 PR 数を確認 → Pending PR を QA エンジニアにアサイン → ワークロードチャートで偏りがないか確認
QA
自分にアサインされた PR を確認 → チェックリストテンプレートを適用 → 項目を順次確認してチェック → 問題があればコメント投稿 → 阻害要因があればブロッカー登録
QA
全チェック項目完了 → ステータスを Testing → Approved に変更 → 承認理由をコメントに記録 → 開発者がマージ可能状態に
Dev
自分の PR のステータスを確認 → Blocked の場合はブロッカー内容を確認 → 修正対応 → QA エンジニアにレビュー再依頼
QA Lead
老朽化 PR アラートを確認 → 7日以上更新されていない PR を特定 → 担当者にフォローアップ → 必要に応じて優先度を再設定
QA Lead
ワークロードチャートでチーム全体の負荷を確認 → 偏りがあれば翌週のアサイン方針を調整 → Blocked PR の根本原因を分析
QA Lead
チェックリストテンプレートを見直し → 頻出する追加チェック項目をテンプレートに反映 → 不要になった項目を削除
Team
週次ミーティングでダッシュボードを画面共有 → ステータス分布の推移を確認 → ボトルネックの特定と対策を議論
Webhook
開発者が GitHub / GitLab で PR を作成 → Webhook イベント発火 → QA Merge Desk に PR が Pending ステータスで自動登録
QA Lead
Pending PR を確認 → QA エンジニアをアサイン → ステータスが In Progress に遷移
QA
チェックリスト適用 → テスト開始 → ステータスが Testing に遷移 → 全項目完了で Approved
QA
テスト中にブロッカー発見 → ブロッカー登録 → ステータスが Blocked に遷移 → 開発者が修正 → ブロッカー解決 → Testing に復帰
