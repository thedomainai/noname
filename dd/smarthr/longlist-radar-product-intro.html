02 — Problem

解決する課題

ロングリストはM&Aパイプラインの最上流にあり、案件の質を決定づける。しかし、ライブディールが増えるほど更新が止まり、将来のパイプラインが痩せていく構造的な問題を抱えています。

100社
初期リスト作成規模
200件/年
仲介経由の紹介数
4-5本
同時並走ライブディール
2-3週
ロングリスト放置期間
探索の停止

ライブディールが重なると、ロングリストの更新が完全に止まる。将来の有望案件の探索が、目先の実行に食われる構造。

情報の陳腐化

放置されたリストの従業員数・事業状況は古くなる。再開時に全件を見直すコストが高く、結局「前と同じ順」で進めてしまう。

判断根拠の揮発

「なぜこの会社を今追うべきか」の理由が記録されず揮発する。担当者が変わると優先順位の根拠が失われる。

突合の手動負荷

仲介からノンネームが届くたびに「これはロングリストにある会社か？」を手作業で照合。年間200件×手動突合。

03 — Deep Dive

課題の構造的分析

表面的な「更新が止まる」の背後には、業務設計上の8つの構造的問題が存在します。これらを正しく捉えなければ、単なるリスト管理ツールに終わってしまいます。

ISSUE 01
投資仮説の「タグ化」

投資仮説を「HR Tech 隣接」「バーティカルSaaS」等のタグに矮小化すると、議論の土台にならない。統括責任者は「通常思いつくようなカテゴライズ程度のものだと全然議論にならない」と明言。投資仮説は設定可能なテーマであり、評価基準・カバレッジ状態を持つべきです。

ISSUE 02
変化の視認性不足

統括責任者の最頻利用パターンは「2-3週ぶりに開く → 何が変わったかだけ見る」。しかし変化情報が通知欄に押し込まれており、主要動線になっていない。開いた瞬間に「前回からの差分」が見える設計が必要です。

ISSUE 03
ノンネーム突合の欠如

PMO実務中核の日次業務で最も高頻度な「ノンネーム → ロングリストにいる会社か？」の突合動線が、既存ツールには存在しない。業種・規模・地域の断片情報から候補を信頼度付きで提示する仕組みが必要です。

ISSUE 04
紹介ルートの非構造化

上位10社へのアプローチは「あの手この手でルートを作り」と語られるが、ルート候補（取引銀行・共通知人・仲介別担当・業界イベント）の管理が構造化されていない。属人的な記憶に依存しています。

ISSUE 05
陳腐化の不可視

「時間がたてばロングリストも鮮度落ちていく」にもかかわらず、陳腐化の程度が視覚的に表現されていない。小さなテキスト（「15日前更新」）では認知負荷が高く、緊急度が伝わりません。

ISSUE 06
エクスポート導線の不在

上位10社を事業部責任者に共有するためのエクスポート機能がない。「なぜこの会社を今追うべきか」の理由付きで出力し、社内の意思決定テーブルに載せる動線が欠落しています。

ISSUE 07
テーマ新規作成フローの欠如

ロングリストは「まず投資仮説（テーマ）を定義する」から始まるが、テーマ定義 → 初期候補生成 → 人手精査のオンボーディングフローが存在しない。リストはあるがゼロイチの構築動線がありません。

ISSUE 08
キャパシティの不可視

「検討したい会社なんだけど検討できません」というライブディール過負荷の状態を可視化する手段がない。新規アウトリーチを控えるべきかの判断が感覚的になります。

失敗コスト: 本来見るべき会社を見逃す。追うべきでない会社に面談工数を使う。上流の案件創出が目先のライブディールに食われ、ロングリスト起点の高打率案件が減る。
04 — Solution

Longlist Radar のアプローチ

ダッシュボードではなく「インテリジェントな秘書」。ユーザーが見ていない間も情報を整理し、根拠付きで答え、今日見るべきものを準備してくれる。意思決定は人間が行い、ツールはその入力を整えます。

01
意思決定を代替しない
結論を自動で出すのではなく、人が判断するための入力を整える
02
根拠が見える
出力には必ずソース・元データとの対応を残す
03
薄く入る
既存の Excel / Google スプレッドシートの上に乗る
04
人手運用を前提に
最初から完全自動でなくてよい。人手確認込みで価値が出る
05
案件差異を扱える
案件ごとの設定変更を前提にした柔軟な構造
「秘書」メタファーの設計含意: 自分が見ていない間も情報を整理してくれている。聞かれたことに根拠付きで答える（判断はしない）。「今日見るべきもの」を準備してくれている。自分の好みや基準を理解している。記録を正確に残してくれる。

As-Is → To-Be

BEFORE
  • 手作業でWeb検索し100社を集める
  • 人手で色分け・優先順位づけ
  • ライブディールが増えると更新が止まる
  • 再開時に情報が古くなっている
  • ノンネーム照合は記憶頼み
  • 上位10社の共有は都度作成
AFTER
  • 既存スプレッドシートに直接接続
  • 投資仮説ベースのスコアリング
  • 公開情報を自動巡回し差分を検出
  • 再開時は「前回からの変化」を5分で把握
  • ノンネームのクイックマッチング
  • 上位N社を理由付きでエクスポート
05 — Features

機能の詳細

5つのコア機能で構成されます。各機能は独立して価値を提供しつつ、組み合わさることでロングリスト運用全体を支える設計です。

FEATURE 01
Longlist Sync
既存の Google スプレッドシートがデータの原本であり続けます。Longlist Radar は独自 DB にデータをコピーせず、スプレッドシートを直接読み書きします。生成した情報（スコア・シグナル・名寄せ結果）は追加列として書き戻します。
  • Google スプレッドシートの URL を指定して直接接続
  • 法人番号（国税庁 API）をキーに企業を名寄せ
  • ユーザーの手動編集を Radar が上書きしない
  • 誤った名寄せを人が修正可能
FEATURE 02
Thesis Config
投資仮説を単なるタグではなく、評価基準とカバレッジ状態を持つ「テーマ」として設定します。テーマの深さが、このプロダクトの差別化ポイントです。
  • 事業領域・規模感・顧客層・除外条件を設定
  • 案件ごとにスコアリングの重みを変更
  • テーマ別カバレッジ状態の可視化
  • テーマ条件から初期候補リストを自動生成
  • 自動生成候補は「未承認」状態で提示、人が承認するとスプレッドシートに追加
FEATURE 03
Signal Update
公開情報を自動巡回し、前回取得時からの差分を検出します。実務担当は差分だけを確認・承認すればよく、100社の全件チェックから解放されます。
  • 従業員数・事業説明・採用動向・資本政策のシグナル検出
  • 前回取得時との差分を保持
  • 差分の承認/棄却ワークフロー
  • 更新日時の自動記録
FEATURE 04
Prioritization
戦略フィット・情報鮮度・接触優先度の3軸でスコアリングし、「なぜ今この会社を追うべきか」を理由付きで表示します。ブラックボックスのスコアは出しません。
  • 3軸スコアの分離表示（戦略/鮮度/接触）
  • 総合スコアの根拠が追える
  • 人が優先度を上書き可能
  • 陳腐化企業の視覚的強調
FEATURE 05
Action Queue
候補企業のステータス管理とアクション導線を提供します。紹介ルートの記録、エクスポート、キャパシティの可視化を含みます。
ステータス管理
未確認 → 調査中 → 紹介打診中 → 面談済 → 保留の状態遷移
接触ルート記録
取引銀行・共通知人・仲介別担当・業界イベントの管理
エクスポート
上位N社を理由・シグナル付きでCSV/PDF出力

意図的に含めないもの

  • CRM化: Salesforceのような顧客管理機能は追加しない
  • 自動アウトリーチ: メール・電話の自動送信は行わない（統括責任者が嫌うもの）
  • バリュエーション自動算出: Longlist Radarの範囲外
  • 仲介向け機能: ツールは社内向けに限定する
06 — User Flow

使い方

Longlist Radarは、日次・週次・月次・四半期の4つの業務サイクルに沿って利用されます。各サイクルでの典型的な操作フローを示します。

初期セットアップ

1
テーマ定義
投資仮説をテーマとして設定。評価基準と重みを定義
2
スプレッドシート接続
既存の Google スプレッドシートに直接接続
3
企業名寄せ
法人番号ベースで名寄せ。揺れ吸収と重複排除
4
初期スコアリング
テーマに基づく戦略フィットスコアを自動算出
5
人手レビュー
スコアと優先順位を確認・上書き。運用開始

業務サイクル別フロー

PMO
ノンネームが届く → ノンネーム照合バーに業種・規模・地域を入力 → ロングリスト上の候補が信頼度付きで表示 → 既知の会社かどうかを30秒で判断
実務担当
Signal Updateが差分を検出 → 差分承認ワークフローで「承認/棄却」を処理 → 確認済みのシグナルが企業情報に反映
統括
Longlist Radarを開く → Weekly Digestビューに自動切替 → 「前回確認から7件の変化がありました」 → シグナルを時系列で確認 → 5分で状況を把握
統括
リストビューに切替 → 優先順位スコアで上位10社を確認 → 各社の「Why Now」理由を読む → 紹介資源の配分を決定
PMO
ステータスフィルタで「調査中」を表示 → 停滞している案件を発見 → アクションを更新または保留に変更
統括
テーマ詳細パネルを開く → テーマごとのカバレッジと空白を確認 → 候補が不足しているテーマを発見 → 探索方向を調整
PMO
鮮度ソートで陳腐化企業を一覧表示 → 更新すべき企業を特定 → 実務担当に更新を依頼
全員
上位10社を選択 → エクスポートボタンで理由・シグナル付きレポートを出力 → 事業部責任者に共有
統括
新しい投資仮説を定義 → Thesis Configで評価基準と重みを設定 → 初期候補の自動生成 → 人手で精査してリスト化
統括
事業環境の変化を受けて → 投資仮説の重みを更新 → スコアが再計算 → 優先順位が自動で入れ替わる
経営陣
パイプラインサマリーを確認 → 追跡社数・フィット分布・陳腐化率でM&A戦略の進捗を把握
