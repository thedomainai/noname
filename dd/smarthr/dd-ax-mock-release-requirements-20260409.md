# SmartHR様 DD AX Mock Release Requirements

更新日: 2026-04-09

## 0. この文書の目的

本書は、各プロダクトについて **HTML/CSS の静的モックをリリースできる状態** を定義するための文書である。

ここでいう「release」は、実装完了ではなく、以下を満たす状態を指す。

1. 社内外の関係者がブラウザで開いて見られる
2. 3〜5分のデモシナリオを、説明なしでもある程度追える
3. 実際の業務に近いダミーデータが入っている
4. 主要なユースケースと主要な状態遷移が、静的画面として再現されている
5. 「このプロダクトは何を置き換え、何を置き換えないか」が伝わる

この文書は、プロダクトマネージャー視点とドメイン理解の両方から、**静的モックを出すまでに必要な内容** を整理する。

## 1. 共通方針

### 1.1 モックの役割

今回のモックは、開発前のデザイン遊びではない。以下の3つを検証するためのものとする。

- `業務の解像度`: 現場が見たときに「そうそう、ここが痛い」と言えるか
- `導入イメージ`: 既存運用のどこに差し込まれるかが具体的に見えるか
- `信頼形成`: AIに意思決定を任せるのではなく、苦労を減らすプロダクトだと伝わるか

### 1.2 モックの成果物

各プロダクトについて、最低限以下の2種類を揃える。

1. `Product Intro HTML`
   - 誰のどの課題を解くか
   - 現状の業務構造
   - どういう考え方で解くか
   - 主要機能
   - デモの見どころ

2. `Application Mock HTML`
   - 実際に使う画面
   - 主要操作の流れ
   - 主要状態
   - リアルなダミーデータ

### 1.3 モックで必ず必要な品質

- `静的でも業務が伝わること`
- `根拠付きのUIであること`
- `ダミーデータが雑でないこと`
- `PC表示を主軸にしつつ、狭幅でも崩れないこと`
- `文言が業務の主語・述語に耐えること`
- `Phase 00 の範囲外をやっているように見せないこと`

### 1.4 デザイン共通要件

- 既存の `Paper DS` 系のトーンを踏襲する
- 画面の主役は派手な演出ではなく、情報構造と比較可能性に置く
- 配色は `paper / slate / ink-blue / ink-amber / ink-green / ink-red` をベースにする
- 書体は既存資産と整合させる
- 数字、日付、スコア、状態、根拠の表示ルールを統一する

### 1.5 共通コンポーネント

全プロダクトで、モック段階でも以下は共通化する。

- ヘッダー
- セクションナビ
- ステータスバッジ
- スコアチップ
- 情報カード
- テーブル
- 右ペイン詳細ビュー
- 根拠リンク / citation 表示
- タイムライン / activity feed
- 空状態 / 注意喚起 / 未確定状態

#### D. 出力プレビュー画面

#### E. 判断ログ画面

## 8. リリース順

### 8.1 優先順位

1. `Longlist Radar`
2. `QA Merge Desk`
3. `DD Navigator`
4. `IM First Pass`
5. `Projection Sentinel`

### 8.2 理由

- `Longlist Radar` は既存資産があり、事業インパクトも説明しやすい
- `QA Merge Desk` は痛みが最も明確で、静的モックに落としやすい
- `DD Navigator` は魅力が強いが、検索・版管理・reading path まで見せないと中途半端になる
- `IM First Pass` は業務理解が要るが、画面数は比較的少なく収められる
- `Projection Sentinel` は価値は高いが、ダミーデータの設計難度が高い

## 9. プロダクト横断で release 前に必要なもの

以下は各プロダクト個別ではなく、全体で必要。

### 9.1 文言品質レビュー

- 主語・述語・目的語の精度
- AIが何をやるか、何をやらないかの線引き
- SmartHR様の所管外テーマを混ぜないこと

### 9.2 ダミーデータ監修

- 会社名、資料名、質問文、論点名、モデル構造が雑でないこと
- ファイル間で整合していること

### 9.3 デモ台本

各モックについて `3分版` と `5分版` の説明順を決めること。

### 9.4 スクリーンショット書き出し

- デスクトップ表示
- 必要に応じてモバイル表示
- 提案資料に貼れるサムネイル

### 9.5 ファイル命名規則

## 10. まとめ

各プロダクトの HTML/CSS モックを出すには、単に1画面を描けばよいわけではない。
必要なのは、以下の4点を揃えることである。

1. `業務上の瞬間` が見えること
2. `その瞬間に必要な情報` が載っていること
3. `主要な状態` が表現されていること
4. `このプロダクトが何を代替せず、何を軽くするのか` が伝わること

この基準で見ると、直近で最も release しやすいのは `Longlist Radar` と `QA Merge Desk`。
次点が `DD Navigator`。
`IM First Pass` と `Projection Sentinel` は、画面を作る前にダミーデータ設計を先にやるのが正しい。
