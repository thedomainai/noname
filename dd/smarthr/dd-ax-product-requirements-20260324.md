# SmartHR様 DD AX Product Requirements

更新日: 2026-03-24

## 0. この文書の目的

本書は、SmartHR様 コーポレート戦略本部における DD AX Phase 00 の候補プロダクトについて、**業務理解に基づく要件定義**を行うための文書である。

対象は以下の5プロダクト。

1. `Longlist Radar`
2. `QA Merge Desk`
3. `DD Navigator`
4. `IM First Pass`
5. `Projection Sentinel`

ここでの要件定義は、単なる機能一覧ではなく、以下の順序で整理する。

1. 業務フロー上の位置づけ
2. 誰が、どの瞬間に、何に困っているか
3. 失敗すると何が起こるか
4. どの体験を最初に作るべきか
5. そのために必要な機能・制約・成功指標は何か

## 1. 共通理解

### 1.1 SmartHR様のDD業務で起きていること

ヒアリングから見えた業務構造は次の通り。

1. ソーシングは `仲介 / 自社ロングリスト / 持ち込み` の3チャネルで進む
2. Pre-DD では IM や基本資料を見て、面談・NDA・簡易 DCF・意向表明の判断を行う
3. 本DDでは `BDD / FDD / LDD / TDD / HRDD / ITDD` など複数モジュールが同時に走る
4. 社内チームは、各モジュールの成果を受け取るだけではなく、PMO、質問の統合、資料共有、論点整理、価格判断まで担う
5. ディールは複数本並行し、Pre-DD と本DDが重なる

つまり、ボトルネックは単一作業ではなく、以下の3種類に分かれている。

- `上流の探索負荷`: ロングリスト作成・更新、IMの初見レビュー
- `実行中の交通整理`: Q&A統合、資料探索、更新通知、バージョン管理
- `終盤の品質担保`: 事業計画・モデルの整合性確認

### 1.2 共通ユーザー像

#### A. 長谷山さん

- 役割: ディール全体を統括する責任者
- 関心: パイプラインの質、重要論点、価格・ストラクチャー判断、チームのレバレッジ
- 嫌うもの: 表層的な提案、黒箱の自動化、所管外の大風呂敷
- 買う理由: 高単価人材の時間を、論点設定と意思決定に戻せること

#### B. 小室さん

- 役割: 実務の中心で PMO・Q&A・案件捌きを担う中核メンバー
- 関心: 重複チェック、送付前品質、案件の初見レビュー、対外印象の維持
- 失敗コスト: セルサイドへの印象悪化、無駄な往復、朝の固定工数

#### C. 平山さん

- 役割: 資料を読み、論点を追い、各モジュールやチームをまたいで理解する実務担当
- 関心: 必要資料への到達、更新把握、参照付き検索、理解の立ち上がり
- 失敗コスト: 資料探索に時間が溶ける、見るべき資料を外す、追加質問が増える

#### D. 外部アドバイザー

- 役割: 各DDモジュールで調査・分析・質問出しを行う
- 関心: 必要資料へのアクセス、質問の優先順位、最新版の共有
- 失敗コスト: 重複質問、古いファイル、論点のズレ

#### E. 経営陣 / 事業部責任者

- 役割: 最終判断や投資仮説の評価を行う
- 関心: 論点の要約、価格への影響、リスクの重大度
- Phase 00 では主な利用者ではないが、最終的な便益の受益者

### 1.3 共通プロダクト原則

Phase 00 では、5プロダクトすべてに次の原則を適用する。

1. `意思決定を代替しない`
   - 結論を自動で出すのではなく、人が判断するための入力を整える
2. `根拠が見える`
   - 出力には必ずソース、元ファイル、元セル、元質問との対応を残す
3. `薄く入る`
   - 既存の Excel / Google スプレッドシート / VDR / Drive の上に乗る
4. `人手運用を前提に始められる`
   - 最初から完全自動でなくてよい。人手確認込みで価値が出ることを優先する
5. `案件ごとの差異を扱える`
   - DDは案件ごとの差が大きい。共通化しつつ、案件ごとの設定変更を前提にする

### 1.4 共通制約

- ローカルAIツールの社内導入は難しい
- いきなり黒箱の自動意思決定は通らない
- VDRや質問表は既存の運用資産として残る前提
- 日本語文書、英語契約書、画像PDF、Excel、Google スプレッドシートが混在する
- 重要なのは「精度100%」ではなく、「高単価人材のレビュー対象を狭めること」

## 2. Product 01: Longlist Radar

### 2.1 位置づけ

対象フェーズは `ソーシング〜Pre-DD前`。

このプロダクトが解くのは、「ロングリストは重要だが、ライブディールが増えると更新が止まる」という問題である。長谷山さんの発言からも、上流の探索は費用対効果が高い一方で、継続運用が難しいことが確認されている。

### 2.2 As-Is 業務フロー

1. 事業部や経営方針を踏まえて対象会社の要件を定める
2. 手作業でWeb検索し、100社前後の候補を集める
3. 人手で色分け・優先順位づけを行う
4. 上位10社程度に対して、紹介ルートや接触方法を探す
5. ライブディールが増えると更新が止まり、しばらく放置される
6. 再開時には、従業員数や事業状況などの情報が古くなっている

### 2.3 コアペイン

- 候補会社の探索と更新が労働集約的
- 情報が古くなっても、更新の優先順位がつけにくい
- どの会社を「今追うべきか」の判断理由が残りにくい
- 既存ディールが重なると、未来の案件パイプラインが痩せる

### 2.4 失敗コスト

- 本来見るべき会社を見逃す
- 追うべきでない会社に時間を使う
- 上流の案件創出が、目先のライブディールに食われる
- ロングリスト起点の高打率案件が減る

### 2.5 JTBD

- When ライブディールで手が埋まっているとき、I want ロングリストの更新だけは自動で維持され、so that 将来の有望案件を取りこぼさないようにしたい
- When 久しぶりにロングリストを見るとき、I want 何が変わったかだけを知り、so that 全件を最初から見直さずに優先順位を付けたい
- When 上位候補を社内共有するとき、I want なぜその会社を今追うべきかを理由付きで示し、so that 紹介打診や面談設定の判断を早くしたい

### 2.6 ユーザーストーリー

- As 長谷山さん, I want 上位候補10社の優先順位と理由を一目で見たい, so that 自分の紹介資源をどこに使うか決められる
- As 小室さん, I want 既存の Google スプレッドシートを直接接続したい, so that 二重入力やファイルのやり取りを一切したくない
- As 小室さん, I want 投資仮説を定義したら初期候補が自動で出てきてほしい, so that 100社を手作業でWeb検索する工程をなくしたい
- As 平山さん, I want 各社の公開情報の更新差分だけを確認したい, so that 全件を調べ直さずに済ませたい

### 2.7 Phase 00 のプロダクト範囲

#### Must Have

- Google スプレッドシートとの直接連携（双方向同期）
- 法人番号ベースの企業名寄せ
- 投資仮説（テーマ）の構造的設定
- テーマ条件に基づく初期候補の自動生成
- 公開情報の差分更新
- 優先順位スコアの表示
- 更新差分の通知
- メモとアクション状態の記録

#### Nice to Have

- 紹介候補や接点候補の半自動提案
- 類似企業の自動追加
- 「今月見るべき10社」の自動生成

### 2.8 機能要件

1. `Longlist Sync`
   - **既存の Google スプレッドシートがデータの原本であり続ける**。Longlist Radar は独自 DB にデータをコピーせず、スプレッドシートを直接読み書きする
   - Google スプレッドシートの URL を指定して接続できること
   - Longlist Radar が生成した情報（スコア・シグナル・名寄せ結果）はスプレッドシート上の追加列として書き戻すこと
   - ユーザーはスプレッドシート上で直接編集を続けられること（Longlist Radar はそれを上書きしない）
   - CSV / Excel からのファイル取込はフォールバックとして対応すること（取込後は Google スプレッドシートに変換して同じ動線に乗せる）

2. `Company Resolution`
   - 法人番号（国税庁 API、無料）をキーに企業を名寄せすること
   - 法人番号で引けない場合は、社名 + 所在地 + 業種の組み合わせで候補を提示すること
   - 名寄せ結果はスプレッドシート上の列（法人番号・正式社名）として書き戻すこと
   - 誤った名寄せを人が修正できること
   - リッチ情報（事業内容・規模・採用動向）は Web クロール + LLM 抽出で補完すること
   - 初期フェーズでは TDB / SPEEDA 等の有料 DB には依存しない

3. `Thesis Config`
   - 事業領域、規模感、顧客層、隣接領域、除外条件を設定できること
   - 案件ごとにスコアリングの重みを変更できること
   - テーマ定義から初期候補リストを自動生成できること（Web クロール + LLM によるスクリーニング）
   - 自動生成された候補は「未承認」状態で提示され、人が承認するとスプレッドシートに行として追加されること

4. `Signal Update`
   - 従業員数、事業説明、採用動向、プロダクト拡張、資本政策関連の公開シグナルを更新できること
   - 前回取得時との差分を保持できること

5. `Prioritization`
   - `戦略フィット`, `情報鮮度`, `接触優先度` を分けて表示できること
   - 総合スコアの理由が分かること

6. `Action Queue`
   - `未確認 / 調査中 / 紹介打診中 / 面談済 / 保留` のような状態管理ができること
   - 上位候補をリストとしてエクスポートできること

### 2.9 非機能要件

- 各スコアの根拠が追えること
- 公開情報の更新日時が残ること
- 誤った名寄せを人が修正できること
- 完全自動ではなく、人が優先度を上書きできること
- 自動生成された候補を人が承認してからリストに入ること
- Google スプレッドシートとの同期で既存データを壊さないこと

### 2.10 非目標

- CRM 全体の置換
- 企業価値算定の自動化
- 経営者への自動営業送信

### 2.11 成功指標

- ロングリスト更新の停止期間を短縮できる
- 上位候補10社の作成時間を削減できる
- 月次の更新対象会社数を維持できる
- ロングリスト由来の面談数を増やせる

## 3. Product 02: QA Merge Desk

### 3.1 位置づけ

対象フェーズは `本DDのPMO運営`。

本DDでは、各アドバイザーが別々に質問表を出す。社内PMOはそれを重複確認し、質問を統合し、セルサイドに送付する。ここは高頻度で、しかも失敗すると対外印象を損なうため、最も素直なクイックウィン候補である。

### 3.2 As-Is 業務フロー

1. BDD / FDD / LDD / TDD 等の各アドバイザーから質問表が Excel で届く
2. PMO が横断で読み、重複・表現揺れ・類似質問を見つける
3. 「残す / 統合する / 削る」を人手で判断する
4. 統合版の質問表をセルサイドに送る
5. 回答が返ってくるたびに、最新版管理と差し戻し対応が発生する

### 3.3 コアペイン

- 質問が重複しやすい
- 類似質問でも、完全一致ではないため単純な突合では足りない
- 朝30分を2名で重複確認する固定工数が発生している
- 一度セルサイドに送ると、誤送や重複が対外印象に直結する

### 3.4 失敗コスト

- セルサイドから「同じことを何度も聞く買い手」と見られる
- PMOが質問統合に時間を取られ、より重要な論点整理に時間を使えない
- モジュール間の論点統合が遅れ、深掘りすべき質問の優先順位が崩れる

### 3.5 JTBD

- When 複数アドバイザーから質問が届いたとき、I want 重複候補を先に洗い出してほしい, so that 自分は統合判断だけに集中したい
- When 類似質問を残すか統合するか迷うとき、I want 元質問と統合候補を並べて見たい, so that 意味の違いを落とさずに整理したい
- When セルサイドに送る前, I want 最終送付版の品質を担保したい, so that 印象悪化を防ぎたい

### 3.6 ユーザーストーリー

- As 小室さん, I want 重複候補を信頼度付きで見たい, so that 全件を一から目視比較しなくて済む
- As 長谷山さん, I want 高優先度の質問だけを上から見たい, so that 本当に重要な論点が埋もれていないか確認できる
- As 外部アドバイザー, I want 自分の質問がどう扱われたか分かる, so that 重要論点が勝手に消えたと感じずに済む

### 3.7 Phase 00 のプロダクト範囲

#### Must Have

- 複数質問表の取込
- 類似質問のクラスタリング
- 統合候補の提示
- 統合版質問表の出力
- 判断ログの保存

#### Nice to Have

- 回答到着後の最新版統合
- モジュール別の質問件数ダッシュボード
- 質問の優先度や期限管理

### 3.8 機能要件

1. `Question Sheet Import`
   - 複数の Excel を取り込めること
   - シート名、行番号、元ファイル、提出元を保持すること

2. `Similarity Engine`
   - 完全一致だけでなく、言い換え・表現揺れ・粒度差を含めて類似判定できること
   - 類似度スコアを表示できること

3. `Merge Workspace`
   - 元質問と統合候補を横並びで見られること
   - `統合`, `残す`, `削除`, `保留` を明示的に選べること
   - 同じ意味でも、期間や対象が違う場合は別質問として残せること

4. `Outbound Export`
   - 統合版質問表を既存運用の形式で出力できること
   - 元質問との対応表を残せること

5. `Decision Log`
   - 誰がどう判断したか、後から追えること

### 3.9 非機能要件

- 誤統合を防ぐため、人が最終承認できること
- 元質問を絶対に失わないこと
- 既存の Excel 運用に戻せること
- 日本語の曖昧な言い換えに一定程度耐えること

### 3.10 非目標

- セルサイドとのやり取り全体の置換
- VDR 自体の置換
- 自動で質問送付まで完結すること

### 3.11 成功指標

- 重複チェック時間を半減できる
- セルサイド送付前の重複率を下げられる
- 質問統合作業の属人性を下げられる

## 4. Product 03: DD Navigator

### 4.1 位置づけ

対象フェーズは `本DDの資料探索と横断理解`。

大量の開示資料が届く本DDでは、読む前に探す時間が長い。加えて、経験の浅いメンバーは「どこから読むべきか」が分からず、必要な論点へ到達するまでの時間が長い。`DD Navigator` は、VDR を置き換えるのではなく、その上に検索・読解レイヤーを追加するプロダクトとして定義する。

### 4.2 As-Is 業務フロー

1. 売り手やFAから資料がVDRに追加される
2. 各メンバーやアドバイザーが、必要な論点に応じて資料を探す
3. どのファイルが最新か、どこに論点が書かれているかを個別に見に行く
4. 経験者は何となく見つけられるが、経験の浅いメンバーは迷う
5. 見つからないと、ベンダーや他メンバーへの追加質問が増える

### 4.3 コアペイン

- 必要な資料と該当箇所に到達するまでの時間が長い
- チームごとに見ている資料が分かれ、共通認識が作りにくい
- 最新版の把握とバージョン差分の追跡が大変
- 経験の浅いメンバーは、何を先に見ればよいか判断しにくい

### 4.4 失敗コスト

- 同じ資料探索を何人も繰り返す
- 追加質問を出す前に社内で確認できたはずの論点を外に聞いてしまう
- 論点共有が遅れ、意思決定者への報告の鮮度が落ちる

### 4.5 JTBD

- When 大量の資料が届いたとき、I want 論点ごとに必要な資料へ最短で到達したい, so that 探索ではなく理解に時間を使いたい
- When 自分が経験の浅い領域を担当するとき、I want 最初に見るべき資料の順番を知りたい, so that 見当違いの読み方を減らしたい
- When チームで同じ論点を議論するとき、I want 同じ根拠資料を見ながら話したい, so that 認識ズレを減らしたい

### 4.6 ユーザーストーリー

- As 平山さん, I want 「資金繰りはどこか」と聞いて該当箇所に飛びたい, so that まず探す時間を減らしたい
- As 小室さん, I want 最新版の資料だけを見たい, so that 古い版を前提に質問を出したくない
- As 長谷山さん, I want 重要論点ごとに参照資料を束ねて見たい, so that 自分で全フォルダを潜らずに済ませたい

### 4.7 Phase 00 のプロダクト範囲

#### Must Have

- 文書取込
- OCR / 全文検索
- 参照付きQ&A
- 論点別関連資料推薦
- 最新版優先表示

#### Nice to Have

- 差分要約
- 新規資料の通知
- 論点ごとの reading path
- 社内メモとの紐付け

### 4.8 機能要件

1. `Document Ingestion`
   - PDF, Office 文書, 画像PDF を取り込めること
   - ファイル名、日付、モジュール、提出元などのメタデータを保持できること

2. `Grounded Search`
   - キーワード検索だけでなく自然文質問ができること
   - 回答には、ファイル名と該当箇所を必ず付けること

3. `Version Awareness`
   - 同名・類似名ファイルの更新版を判定し、最新版を優先表示できること
   - 旧版も辿れること

4. `Issue-Based Navigation`
   - `ネットデット`, `主要顧客`, `資金繰り`, `CoC条項` のような論点ごとに、関連資料を束ねて見られること
   - 経験の浅いメンバー向けに「まずこの3資料を見る」を提示できること

5. `Shareability`
   - 検索結果や回答を、チーム内へ共有できること

### 4.9 非機能要件

- 幻覚的な回答を避けるため、根拠がない場合は「不明」と返せること
- OCR の誤りがあっても元PDFへ戻れること
- 高速に検索できること
- 既存VDRの権限管理を壊さないこと

### 4.10 非目標

- VDRそのものの置換
- 売り手・買い手・FA の三者向けコラボレーション基盤の実装
- ボード資料の自動完成

### 4.11 成功指標

- 資料探索にかかる時間を削減できる
- 追加質問前の社内自己解決率を上げられる
- ジュニアメンバーの立ち上がり時間を短縮できる

## 5. Product 04: IM First Pass

### 5.1 位置づけ

対象フェーズは `IM受領〜Pre-DDの初期判断`。

SmartHR様では年間200件程度の紹介があり、そのうち40〜45件程度の IM を閲覧している。ここでは「全部を深く見る」ことは不可能であり、初見の捌き方が案件パイプライン全体の質を決める。

### 5.2 As-Is 業務フロー

1. 仲介や紹介経由でノンネーム / IM が届く
2. ざっと読んで、箸にも棒にもかからないものを落とす
3. 少し可能性があるものは、事業部や責任者と議論する
4. 面談に進むか、NDA に進むかを決める
5. 場合によっては簡易DCFやマルチプル確認に進む

### 5.3 コアペイン

- 流入量に対して、全件を同じ深さで見ることはできない
- 担当者ごとに見る観点の粒度がぶれやすい
- 「今は判断できない未確定要素」と「明確な赤旗」が混ざりやすい
- 深い投資仮説が価値の源泉なので、一般論の要約では意味がない

### 5.4 失敗コスト

- 有望案件を早い段階で落としてしまう
- 明らかに薄い案件に面談工数を使う
- 事業部やCFOとの初期議論で、論点の前提が揃わない

### 5.5 JTBD

- When IM が届いたとき、I want 投資仮説に照らした一次評価を短時間で作りたい, so that 面談やNDAに進む価値があるかを早く判断したい
- When 社内で案件を共有するとき、I want 何が分かっていて何が未確定かを分けて示したい, so that 議論の論点を早く揃えたい
- When 有望そうだが確信がない案件を扱うとき、I want 次に何を聞けば判断できるかを知りたい, so that 面談準備を短時間で済ませたい

### 5.6 ユーザーストーリー

- As 小室さん, I want IMごとに同じ型の一次メモを作りたい, so that 案件比較をしやすくしたい
- As 長谷山さん, I want 一般論ではなく「この案件で何が論点か」を最初から見たい, so that 自分のレビュー時間を圧縮したい
- As 事業部責任者, I want 面談前に確認すべき論点を知りたい, so that 初回面談の質を上げたい

### 5.7 Phase 00 のプロダクト範囲

#### Must Have

- IM / ノンネーム / 基本資料の取込
- 一次評価メモの生成
- 赤旗論点と未確定論点の分離
- 面談で聞くべき質問の草案
- Go / Watch / Pass の仮ラベル

#### Nice to Have

- Longlist Radar 上の企業データとの接続
- 類似案件比較
- 簡易DCF / マルチプル確認の下書き

### 5.8 機能要件

1. `Deal Pack Parsing`
   - IM, ノンネーム, 会社紹介資料を取り込めること
   - 会社概要、事業内容、顧客、収益構造、成長率、オーナー意向などを抽出できること

2. `Thesis-Based Memo`
   - SmartHR様の投資仮説に沿って、`合う理由`, `懸念点`, `未確定事項` を分けて出せること
   - ありきたりな業界要約に終わらないこと

3. `Next Questions`
   - 次回面談で聞くべき質問を出せること
   - 「この質問に答えが出れば Go / Pass 判断が進む」という形で並べられること

4. `Decision State`
   - `Go / Watch / Pass / Unknown` を保持できること
   - なぜそのラベルなのか理由を残せること

### 5.9 非機能要件

- 黒箱スコアだけを出さないこと
- 投資仮説の重みを案件ごとに変更できること
- 未知の要素を未知のまま扱えること
- 「判断保留」が自然に選べること

### 5.10 非目標

- 買収判断の自動化
- フルバリュエーションの自動生成
- 業界知見の完全自動抽出

### 5.11 成功指標

- IM 初見レビューにかかる時間を削減できる
- 一次メモの比較可能性を高められる
- 面談前の質問準備時間を短縮できる

## 6. Product 05: Projection Sentinel

### 6.1 位置づけ

対象フェーズは `本DD終盤〜価格判断前`。

長谷山さんの発言から、事業計画モデルでは `合計の誤り`, `参照範囲の誤り`, `意味合いの誤り`, `二重計上`, `シナリオ反映漏れ` が典型的な痛点であることが分かっている。ここは価値が大きいが、初手から自動補正を狙うべきではない。`危ない箇所を漏れなく洗い出す` 補助プロダクトとして定義する。

### 6.2 As-Is 業務フロー

1. 事業計画モデルが Excel / Google スプレッドシートで作られる
2. 担当者とレビューアーが、月次列・四半期列・シナリオ列を目検する
3. 合計、参照、意味、前提の整合性を人手で確認する
4. 忙しい終盤では、全量確認が難しくなる
5. 重要なエラーが埋もれたまま価格議論に進むリスクがある

### 6.3 コアペイン

- シート構造が案件ごとに違う
- 構造エラーと意味エラーが混在する
- 人間は全部を見ると疲れ、危険箇所に集中しづらい
- 色やセル装飾で意味を持たせている場合がある

### 6.4 失敗コスト

- 誤った価格レンジや感応度を前提に議論してしまう
- 終盤での修正コストが大きい
- 忙しい局面ほど、高単価人材が目検に貼り付く

### 6.5 JTBD

- When 終盤で複雑なモデルをレビューするとき、I want 危ない箇所だけを先に洗い出したい, so that 自分はそこだけを重点的に確認したい
- When 決定論的なエラーと意味上の違和感が混ざるとき、I want 種類を分けて見たい, so that どこから潰すべきか判断したい
- When 誤検知が出るとき、I want 意図した設計として除外できる, so that 次回以降のノイズを減らしたい

### 6.6 ユーザーストーリー

- As 長谷山さん, I want 価格議論の前に危険アラート一覧を見たい, so that 全シートを最初から追わずに済ませたい
- As 実務担当者, I want どのセルがどの理由で危ないか知りたい, so that 修正の優先順位をつけたい
- As レビューアー, I want 誤検知を除外理由付きで閉じたい, so that 同じノイズを繰り返し見たくない

### 6.7 Phase 00 のプロダクト範囲

#### Must Have

- Excel / Google スプレッドシートの読込
- 決定論的チェック
- 推論的チェック
- アラート一覧
- 元セルへのジャンプ
- 誤検知の除外

#### Nice to Have

- テンプレート別ルールセット
- シナリオ比較の自動差分
- バリュエーション影響の簡易表示

### 6.8 機能要件

1. `Workbook Parsing`
   - シート構成、セル値、数式、参照関係を読み取れること
   - Google スプレッドシートと Excel の両方に対応できること

2. `Deterministic Checks`
   - SUM 範囲ずれ
   - 月次と四半期の不整合
   - 参照切れ
   - 隠れた空欄やエラー値
   - 一貫しない数式パターン

3. `Heuristic Checks`
   - 二重計上の疑い
   - シナリオ適用漏れの疑い
   - 説明変数と結果の不自然な飛び
   - 意図不明な入力切替

4. `Review Workspace`
   - アラートを `構造`, `計算`, `意味`, `前提` で分類できること
   - 重大度と信頼度を表示できること
   - 元セルと影響先セルを辿れること
   - `誤検知`, `対応済み`, `要確認` を付けられること

5. `Rule Config`
   - 案件ごとに「この列は四半期」「この色は入力セル」などの補助ルールを設定できること

### 6.9 非機能要件

- 自動修正しないこと
- 誤検知があっても、元データを壊さないこと
- シート構造の違いにある程度耐えること
- 日本語の行見出しでも扱えること

### 6.10 非目標

- 価格決定の自動化
- 事業計画そのものの作成
- 経営判断の推奨出力

### 6.11 成功指標

- 典型的な整合性エラーを検知できる
- レビュー対象セル数を絞り込める
- 終盤レビューの工数を削減できる

## 7. プロダクト横断の優先順位

### 7.1 Phase 00 で先に作るべき順

1. `Longlist Radar`
2. `QA Merge Desk`
3. `DD Navigator`
4. `IM First Pass`
5. `Projection Sentinel`

### 7.2 理由

- `Longlist Radar` は機密データ依存が小さく、上流の経営インパクトが大きい
- `QA Merge Desk` はペインが最も明確で、対外印象の維持にも効く
- `DD Navigator` は PMO とジュニア支援の両方に効き、将来の基盤にもなりやすい
- `IM First Pass` は有望だが、投資仮説のチューニングが必要
- `Projection Sentinel` は価値は大きいが、技術的難度と案件依存性が最も高い

## 8. まとめ

この5プロダクトは、すべて「高単価人材がやるべきでない苦労を減らし、意思決定のための時間を取り戻す」という同じ戦略に乗っている。

ただし、業務の重さと導入障壁は同じではない。

- `Longlist Radar` は上流の探索負荷を減らす
- `QA Merge Desk` はPMOの重複確認を減らす
- `DD Navigator` は資料探索と理解立ち上がりを速める
- `IM First Pass` は初見レビューの型を揃える
- `Projection Sentinel` は終盤の品質確認を重点化する

Phase 00 では、まず `Longlist Radar` と `QA Merge Desk` で信頼を取り、次に `DD Navigator` を基盤に広げていくのが最も筋が良い。
