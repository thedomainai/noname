# Longlist Radar — PM分析: ユーザー・ユーザーストーリー・UI改善提案

更新日: 2026-03-30

## 1. ユーザー定義（利用場面の解像度で定義する）

### 1.1 長谷山さん — ディール統括責任者

**日常の行動パターン**:
- 週1回程度、パイプライン全体を俯瞰する「棚卸し」を行う
- ライブディールが4-5本並走すると、ロングリストに触れる余裕がなくなる
- 再開時に「何が変わったか」だけを知りたい。全件を見直す気はない
- 上位10社の追い方を決める際、紹介ルートの構築に自分の人脈を使う
  - 具体的なルート: 取引銀行、Facebook、業界知人、仲介会社の別担当
  - 「あの手この手で会えないかっていうルートを作り」（33:19）
- 投資仮説の深さにこだわる。「通常思いつくようなカテゴライズ程度のものだと全然議論にならない」（長谷山）

**この人が嫌うもの**:
- 浅いカテゴリで企業を並べただけのリスト
- 根拠なしのスコアリング
- 自分の判断を飛ばして何かを自動送信するツール

**この人がツールに求めること**:
- 2-3週間触れなくても、再開時に5分で「今追うべき会社」がわかること
- 紹介ルートの記録と候補が手元にあること
- 投資仮説を深く設定でき、その仮説に照らしたフィット理由が言語化されていること

### 1.2 小室さん — PMO実務中核

**日常の行動パターン**:
- 仲介経由のノンネーム/IMを日常的に受け取る（年間200件 = 営業日あたり約1件）
- ノンネームが届いた時に「これ、ロングリストに既にある会社では？」と突合する
- ロングリストの初期作成時、100社程度をWeb検索で集める「労働集約的」な作業を担う
- 候補企業の色分け・優先順位づけの第一稿を作る
- 仲介経由で来た案件の初見レビューも行う（IM First Passとの接点）

**この人が嫌うもの**:
- 二重入力（既存のGoogleスプレッドシートに加えて別のツールにも入力する）
- 手作業で100社を検索して集める時間
- ノンネームが来るたびに、ロングリストを手で突合する作業

**この人がツールに求めること**:
- 既存スプレッドシートからの一発取込
- ノンネーム → ロングリスト候補のクイックマッチング
- 新たな候補企業の半自動追加（検索→候補提示→承認）

### 1.3 平山さん — 実務担当

**日常の行動パターン**:
- ロングリスト上の個社について、公開情報を定期的に更新する
- 採用動向、プレスリリース、資本政策、プロダクト変更を追いかける
- 更新した情報を「どこが変わったか」のdiffで共有する
- DD本体と並行して動くため、ロングリスト更新は後回しになりがち

**この人が嫌うもの**:
- 100社分の情報を1社ずつ手で調べ直す作業
- 前回との差分がわからない状態で全件レビューすること

**この人がツールに求めること**:
- 公開情報の自動巡回と差分検出
- 差分だけを確認して承認/棄却する作業フロー
- 個社の情報更新履歴（タイムライン）

### 1.4 ユーザーの「ユーザー」— 社内ステークホルダー

ロングリストの直接利用者は上記3名だが、その先に以下の受益者がいる。

**事業部責任者**:
- 「この会社を追いたい」という提案を受け取り、面談や紹介の判断をする
- ロングリストの上位10社レポートが、この人への共有資料になる
- 求めるもの: なぜ今この会社を追うべきかの理由、会社の基本情報の要約

**経営陣（CEO/CFO）**:
- M&Aパイプラインの質と量を四半期単位で確認する
- 求めるもの: パイプラインの健全性指標（追跡社数、フィット分布、陳腐化率）

## 3. 現在のモックの構造的問題

### 3.1 投資仮説が「タグ」に矮小化されている

**問題**: 現在のモックでは投資仮説が「HR Tech 隣接」「バーティカルSaaS」等のタグとして実装されている。これは長谷山さんが明確に否定した「通常思いつくようなカテゴライズ程度」そのものである。

**あるべき姿**: 投資仮説は設定可能な「テーマ」であり、各テーマに以下の属性を持つ:
- テーマの名称と説明（例: 「建設業の労務DX — 建設業の人事・安全管理デジタル化で、SmartHRの建設業展開を加速する」）
- スコアリングに影響する評価基準（例: 建設業顧客比率、API連携能力、現場作業員の定着率）
- テーマ内の候補企業の一覧
- テーマのカバレッジ状態（候補が十分か、空白があるか）

### 3.2 「今週の変化」が右サイドバーに押し込まれている

**問題**: 長谷山さんの最も頻繁な利用場面は「2-3週間ぶりにロングリストを開く → 何が変わったかだけ見る」である。現在のモックでは右サイドバーの通知欄に追いやられており、主要動線になっていない。

**あるべき姿**: Weekly Digestを独立したビュー（タブまたはトグル）として昇格させる。開いた瞬間に「前回確認からの変化」が時系列で見えるべき。

### 3.3 ノンネーム突合機能がない

**問題**: 小室さんの日次業務で最も頻度が高い「ノンネームが来た → ロングリストにいる会社か？」の突合動線がない。

**あるべき姿**: ヘッダーに「ノンネーム照合」ボタンまたは検索バーを配置。業種・規模・地域などの断片情報を入力すると、ロングリスト上の候補企業を信頼度付きで提示する。

### 3.4 紹介ルートの管理がない

**問題**: 長谷山さんの発言「上位10社ぐらいにあの手この手で会えないかっていうルートを作り」を支援する機能がない。詳細モーダルに「紹介ルート検索」ボタンがあるがダミー。

**あるべき姿**: 各企業のDetailパネルに「接触ルート」セクションを設け、ルート候補（取引銀行、共通知人、仲介会社の別担当、業界イベント等）を記録・検索できるようにする。

### 3.5 陳腐化の視覚表現が弱い

**問題**: 「時間がたてばロングリストも鮮度落ちていく」が、現在は小さなテキスト（「15日前更新」）でしか表現されていない。

**あるべき姿**: カードの背景色またはボーダーを使って、陳腐化を視覚的に強調する。14日以上更新がない企業はカード全体が視覚的に退色し、ユーザーの注意を引く。

### 3.6 エクスポート機能がない

**問題**: 「上位10社を事業部責任者に共有する」ためのエクスポート導線がない。

**あるべき姿**: ソート/フィルタ結果の上位N社を、理由・シグナル付きで出力できるボタン。CSV/PDF/Googleスプレッドシート連携。

### 3.7 テーマ新規作成の導線がない

**問題**: ロングリストの作成は「まずテーマ（投資仮説）を定義する」から始まる。現在のモックはリスト画面しかなく、テーマ定義→初期候補生成→人手精査のオンボーディングフローが欠落している。

### 3.8 キャパシティの可視化がない

**問題**: 「検討したい会社なんだけど検討できません」というライブディール過負荷の状態を可視化する手段がない。

**あるべき姿**: ヘッダーまたはサマリーに「ライブディール数」とチームキャパシティの指標を表示。パイプラインが過負荷の時に、新規アウトリーチを控えるべきか判断できる。

## 4. UI改善提案（優先度順）

### P0: 必ず反映すべき（モックの説得力に直結）

#### 4.1 ビューモードの追加: Weekly Digest

カード一覧の上部に **「リスト」「今週の変化」** のビュータブを追加する。

「今週の変化」ビューでは:
- 企業ごとにグルーピングせず、シグナル（変化イベント）の時系列で表示
- 各シグナルから元企業カードへワンクリックで遷移
- 「前回確認日」を記録し、前回以降の変化だけをフィルタ

理由: 長谷山さんの最頻利用パターンに直結。

#### 4.2 投資仮説の設計画面

左サイドバーの「投資仮説」セクションを再設計:
- タグをクリックするとフィルタだけでなく、テーマの詳細パネルが開く
- テーマの説明文、評価基準、候補企業数、カバレッジ状態を表示
- 「+ 新テーマ作成」ボタンでテーマ定義→初期候補生成の導線を提供

理由: 投資仮説の深さが商品の差別化ポイント。

#### 4.3 陳腐化の視覚的強調

14日以上更新なしのカードに対して:
- 左ボーダーをグレー破線に変更
- カード全体の不透明度を下げる（opacity: 0.7）
- 「⚠ 情報陳腐化」バッジを表示

理由: 「鮮度が落ちていく」問題の体感化。

### P1: 高優先度（実用性に影響）

#### 4.4 ノンネーム照合バー

ヘッダーに検索バーを追加。「業種: HR Tech、従業員: 100-200名、地域: 東京」のような断片情報を入力すると、ロングリスト上のマッチ候補を信頼度付きで表示。

理由: 小室さんの日次業務（年間200件の処理）を直接支援。

#### 4.5 接触ルートセクション（Detail内）

Detail モーダルに「接触ルート」タブを追加:
- ルートの種類: 取引銀行 / 共通知人 / 仲介別担当 / 業界イベント / SNS
- ルートごとの状態: 未着手 / 打診中 / 応答あり / 不成功
- メモ欄

理由: 「あの手この手でルートを作り」を構造化する。

#### 4.6 上位N社エクスポート

フィルタ/ソート後のリスト上部に「上位10社をエクスポート」ボタンを配置。出力内容:
- 企業名、業種、規模、フィットスコア、Why Now理由、直近シグナル
- PDFまたはスプレッドシート形式

理由: 事業部責任者への共有が必要なワークフロー。

### P2: 中優先度（体験の厚みに影響）

#### 4.7 キャパシティインジケーター

ヘッダー右側に「ライブディール: 3本 / キャパ: 5本」のような表示。
過負荷（4/5以上）時に色が変わり、新規アウトリーチの可否を暗示する。

#### 4.8 テーマ別カバレッジマップ

左サイドバーの投資仮説に、テーマごとの候補社数とステータス分布のミニチャートを追加。
空白テーマ（候補が少ない）を視覚的に目立たせる。

#### 4.9 差分承認ワークフロー

「今週の変化」ビューで各シグナルに「確認済み」チェックを入れられるようにする。
全シグナルを確認すると「前回確認日」が更新される。

## 5. 改善しないべきこと（Non-Goals）

以下は意図的に含めない:
- **CRM化**: Salesforceのような顧客管理機能は追加しない。ロングリストに特化する
- **自動アウトリーチ**: メール・電話の自動送信機能。長谷山さんが嫌うもの
- **バリュエーション自動算出**: Longlist Radarの範囲外（Projection Sentinelの領域）
- **仲介向け機能**: ツールは社内向け。仲介会社へのポータル共有はしない

## 6. 「秘書」メタファーの設計含意

長谷山さんの発言「AIっていうよりはやっぱ秘書さんにお願いするみたいな感じ」は重要な設計示唆である。

**秘書的なツールの特徴**:
1. 自分が見ていない間も情報を整理してくれている
2. 聞かれたことに根拠付きで答える（判断はしない）
3. 「今日見るべきもの」を準備してくれている
4. 自分の好みや基準を理解している（投資仮説のコンテキスト）
5. 記録を正確に残してくれる

これは「ダッシュボード」ではなく「インテリジェントアシスタント」のUIを意味する。
現在のモックは「ダッシュボード」寄りなので、以下を追加すると秘書的になる:
- 起動時に「前回から〇件の変化がありました」のサマリーカード
- ソート結果の上に「今週のおすすめアクション: XX社に紹介打診の適期です」のような能動的提案
- Detailパネルの冒頭に「前回確認（3/20）以降の変化: 2件」の要約

## 7. モック更新の実装方針

上記のうちP0（4.1〜4.3）は今回のモック更新で実装する。P1以降はモック上ではボタン/プレースホルダーとして存在を示す。

### 実装する変更:
1. ビュータブ（リスト / 今週の変化）の追加
2. 投資仮説タグをクリックするとテーマ詳細パネルが開く設計に変更
3. 陳腐化カードの視覚的退色
4. ヘッダーにノンネーム照合検索バーを追加（プレースホルダー）
5. Detail内に接触ルートセクションを追加
6. 上部にサマリーメッセージ（「前回から7件の変化」）を追加
7. 上位N社エクスポートボタン
