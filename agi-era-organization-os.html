第1部 — 全体概念地図
OS Framework

AGI/ASI時代のOS — 10レイヤー定義

知能の能力が上がるほど、「誰のためか」「誰が決めるか」「誰が利益を得るか」がOSの本体になる。認知アーキテクチャは中下位レイヤーに過ぎない。

Domain A — L1〜L3
組織は何者か
存在理由と統治原理
AGI/ASI段階でボトルネックになる最上位領域。能力が高まるほどここが支配的になる。
Domain B — L4〜L6
組織はどう生きるか
Lived Reality（人間の日常）
上位理念を人間のリアリティに落とし込む変換層。ここが弱いと理念は現場で壊れる。
Domain C — L7〜L10
組織はどう知り、行動し、進化するか
知能的実装と変化能力
既存のAI組織論の多くがここだけを扱う。しかし上位なき下位最適化はOSにならない。

能力が低い時代には下位レイヤーがボトルネックになる。能力が高い時代には上位レイヤーがボトルネックになる。AGI段階ではL5〜L8が前景化し、ASI段階ではL1〜L3とL10が支配的になる。
OSを設計するときの5原則
  • 上位レイヤーなき下位最適化を禁止する — 認知・エージェント・実行だけを設計してもOSにはならない
  • 人間は「残余タスク処理者」ではなく設計対象そのもの — 主体性・熟達・成長・意味・関係性を設計する
  • 反省は出力改善だけでなく制度改善へ拡張 — ポリシー・権限・文化設計の修正まで含める
  • 分配の設計を後回しにしない — 分配はOSの最初から含めるべき後工程ではない
  • 速度より正当性、正当性より更新可能性 — 長期的に「正当性があるか」「改定できるか」が重要

典型的な失敗パターン
Level 3をLevel 4として扱う
Symphony等の業務AIを「会社全体のAI」と位置づけると、統治なき自律化が進む。複数業務AIが矛盾した判断を下しても調停できない。
Level 4のないLevel 3の増殖
部署ごとに業務AIを導入するが、全社の目的・価値・禁則との整合を誰も担保しない。「高速なAIの上に低速な官僚制を載せたもの」になる。

第3部 — 目標地点の詳細設計図
Implementation Design

Level 4 実装設計 — Neural Organization OS

AIが標準運転し、人間が目的・価値・境界・例外・関係・進化を担う組織。管理をプロセスから憲法へ移すことが中核思想。

このOSを駆動する6原理
Purpose first
KPIは目的の代理変数にすぎない。目的・非目的・禁則を先に定義する
Governance by design
承認は後付けの会議ではなく、権限・リスク・評価に埋め込む
AI as default operator
標準作業はAIが回す。人間は例外と進化を担う
Human as constitutional authority
人間は手順を実行するのでなく、手順が従うべき原理を定める
Selective human intervention
常時HITLではなく、HUTLを標準にし、HOTL/HITLを条件付きで使う
Agency preservation as a KPI
人間が意味ある異議申立て・停止・再定義をできること自体を性能指標にする
OSの上位構造（5コンポーネント）
A — Purpose Council
目的評議会
人間中心
  • 組織の存在理由を定義
  • 価値・禁則の策定
  • ステークホルダー順位
  • 資本配分原理
  • 原則として人間が主権
B — Governance Kernel
統治カーネル
人間優位
  • リスク分類・Trust Score
  • Composite Risk算出
  • 承認ポリシー
  • 権限・停止条件
  • 監査基準・判例
C — Domain Cells
業務ドメインセル
人間 + AI
  • Domain Governor（人間）
  • AIエージェント群
  • Product / GTM / Finance
  • Legal / People / Ops
  • 各セルが境界・評価基準を持つ
D — Execution Fabric
実行ファブリック
AI中心
  • Symphony型実行エージェント
  • Data Agent型分析エージェント
  • 業務自動化エージェント
  • 実際の作業の主戦場
  • Level 3が稼働する層
E — Reflection Engine
反省・進化エンジン
AI中心・人間裁定
  • 評価・インシデント分析
  • メモリ更新
  • Trust Score更新
  • ポリシー改訂提案
  • 最終裁定は人間が持つ

業務フロー — 8ステップ（Sense → Evolve）
Step 0
Constitution Load
AIがPurpose / Values / Non-negotiables / Risk policy / Tool permissionsを読み込む。HUTLの本体。
HUTL基盤
Step 1
Sense
新しいsignalを受け取る（苦情・売上低下・バグ・規制変更）。AIがeventとして正規化。
AI主導
Step 2
Understand
AIがdomain graphとmemoryを引き、世界状態を再構成。不足文脈はActive Extractionで取得。
AI主導
Step 3
Deliberate
AIが複数案を作成。各案に前提・期待便益・副作用・失敗条件・rollback案・必要権限を付ける。
AI主導
Step 4
Gate
Autonomy Mode = f(Trust Score, Composite Risk, Novelty, Uncertainty, Agency Impact)を計算。
判定
Step 5
Act
canary rollout / rate limit / kill switch / rollbackを必須にして実行。権限・行動範囲・可逆性で統制。
実行
Step 6
Observe
結果だけでなく副作用と二次影響も観測（顧客離反・問合せ増・監査フラグ・チーム負荷）。
AI主導
Step 7
Reflect
4軸評価: Output accuracy / Assumption validity / Value alignment / Agency preservation。
評価
Step 8
Evolve
Memoryを更新し、Trust Scoreを更新し、Policy改訂案を作る。「学習してよいこと」は人間が決める。
人間裁定
Autonomy Mode — 6段階
A0
Sandbox Only
本番禁止。テスト環境のみ。
A1
Full Auto
内部・可逆・低リスクの完全自律実行。
A2
Auto + Post-audit
自動実行するが事後監査が必須。
A3
HOTL
監視者にveto windowを与える。
A4
HITL
事前承認が必要。人間がゲート。
A5
Human-only
AIは提案のみ。実行は人間。
Type 業務の性質 例 対応Autonomy Mode
Type 1 低リスク・可逆・内部作業 定型分析・レポート生成・コード修正 A1 / A2（完全自律または事後監査）
Type 2 低〜中リスク・判断支援 需要予測・価格案・採用スクリーニング A3（AI主導 + HOTL / veto可能）
Type 3 外部影響あり・中〜高リスク 契約提案・価格変更・重要顧客対応 A4（HITL / AIが作り人間が裁定）
Type 4 価値衝突・不可逆・権利影響 人事処分・解雇・法的争点 A5（Human-led / AIは選択肢のみ）
Type 5 危機・未知・制度改定 インシデント・炎上・規制変更 Human command mode（A5強制）
データアーキテクチャ — 6面構造（組織世界モデル）

このOSは単なるRAG付きチャットでは動かない。6面構造の組織世界モデルが必要。

Plane 1
Reality Plane
ERP/CRM/プロダクトログ/会話/市場データ等の現実データ。immutable event logで保持。
Event Log
Plane 2
Semantic Plane
顧客・契約・案件・機能・規制等のエンティティと関係。Knowledge Graphを中心に置く。
Knowledge Graph
Plane 3
Memory Plane
Semantic / Episodic / Procedural / Evaluative / Normative の5種類の組織記憶を格納。
5種記憶
Plane 4
Decision Plane
Goal・仮説・前提・シミュレーション・候補案・選択理由を保持。「思考の可観測化」。
Decision Ledger
Plane 5
Action Plane
ツール実行・APIコール・通知・デプロイ等。Workflow stateとrollback planを持つ。
Action Log
Plane 6
Governance Plane
権限・ポリシー・Trust Score・Composite Risk・監査証跡。OSのcontrol plane。
Control Plane
運用リズム
頻度活動主体
常時 Sense / Understand / Act / Observe AI
毎日 Exception Councilが高リスク案件を裁く 人間
毎週 Domain GovernorがpolicyとevalをUpdate 人間
毎月 Purpose Councilが資源配分と大方針を見直す 人間
四半期 Purpose / Values / Non-goalsを再定義 人間
定例会議の中心を「状況共有」から「例外裁定と制度更新」へ移す。状況共有はAIが常時出せるため、会議でやる必要はない。
